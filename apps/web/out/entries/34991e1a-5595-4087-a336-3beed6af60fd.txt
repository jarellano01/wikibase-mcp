28:T14f5,# gdbt CLI: Evolution from Bash Wrapper to Python Toolchain

## Problem It Solved

Running `dbt` directly had a fundamental issue: dbt doesn't load `.env` files automatically, and the project required environment variables (Snowflake credentials, row limits, feature flags) to be present before any dbt command executed. Engineers were manually exporting variables or using `source .env && dbt run`, which was error-prone and inconsistent with the VS Code / Cursor dbt Power User extension.

The `CustomDbtRunner` Python class was created first — it wraps dbt's own `dbtRunner`, loading `.env` via `python-dotenv` before any invocation. This also made the Power User extension work correctly by pointing it at the custom runner.

## Phase 1: Bash Wrapper (Dec 2025)

**DATA-4114** (Dec 10, 2025) — the first `gdbt` commit.

A bash script named `gdbt` installed to the user's path via a `make install-gdbt` target. Its only job: `cd` into the correct dbt project directory, then delegate to `dbt_custom_runner.py`. The custom help output when run with no arguments was added in the same week.

```bash
# What gdbt did at its simplest:
cd /path/to/dbt_project
python scripts/shared/dbt_custom_runner.py "$@"
```

The key insight: by `cd`-ing first, the `CustomDbtRunner` could reliably find `.env` in `Path.cwd()` regardless of where the engineer ran the command from.

## Phase 2: Commands Beyond dbt (Jan–Feb 2026)

`gdbt` grew a subcommand system. Commands that had nothing to do with raw `dbt` execution began shipping:

| Command | Added | Purpose |
|---------|-------|---------|
| `gdbt validate` | Jan 2026 | Run all validations + auto-fix + regenerate codegen |
| `gdbt validate --ci` | Jan 2026 | Fail if any files modified (CI enforcement) |
| `gdbt validate --force` | Feb 2026 | Full rewrite of all codegen sections |
| `gdbt lint` / `gdbt lint-fix` | Jan 2026 | Run sqlfluff independently |
| `gdbt validate-task-groups` | Jan 2026 | Validate YAML configs only |
| `gdbt dag-pipeline` | Feb 2026 | Interactive pipeline generator |
| `gdbt regenerate-dags` | Feb 2026 | Rebuild all Airflow Python DAGs from YAML |

**DATA-4428** consolidated pipeline validation into `scripts/shared/` — the same scripts now power `gdbt validate`, the CI action, and can be run standalone.

## Phase 3: Modularization (Feb 2026)

**DATA-4514** (Feb 17, 2026) broke the monolithic bash/Python hybrid into proper Python modules under `scripts/shared/`:

```
scripts/shared/
├── dbt_custom_runner.py      # dbt invocation with .env loading
├── gdbt_config.py            # Project config system
├── generate_dag.py           # DAG file generation from YAML
├── generate_pipeline.py      # Pipeline/task group processing
├── generate_task_group.py    # Task group code generation
├── project_config.py         # Multi-project configuration
├── run_custom_validations.py # Validation orchestration
├── snowflake_utils.py        # Snowflake query helpers
├── clone_database.py         # DB clone for local dev
└── cache_utils.py            # Setup cache management
```

Also in this phase: `gdbt` gained **git-diff-aware validation** — it only regenerates files affected by changed models rather than doing a full rewrite, making validation 3–5x faster on typical PRs.

## Phase 4: Multi-Project & Portability (Mar 2026)

**DATA-4567** (Mar 2026) was the biggest quality-of-life upgrade:

- **Portable installation** — `gdbt` resolves all script paths relative to its own install location, not the current directory. You can run `gdbt validate` from anywhere in the repo.
- **Multi-project support** — a single `gdbt` install can target both `dbt_analytics` and `dbt_aiml` via config
- **`gdbt quickstart`** — automated first-time setup: Python environment, `.env` generation, DB clone, dbt dependency install
- **`gdbt reset`** — tear down local environment back to a clean state (preserves `.python-version`)

## Skills Are Files

Each skill is a `SKILL.md` file stored in `.claude/skills/<name>/SKILL.md`. The file has YAML frontmatter followed by the full prompt:

```yaml
---
name: triage
description: Analyze a GitHub issue and propose an implementation approach
allowed-tools: Read, Glob, Grep, Bash(gh:*), Bash(go:list)
---

You are a senior engineer on the Cronos project...
[full prompt follows]
```

The skill file IS the prompt — loaded verbatim at runtime. This makes skills:
- Human-readable (anyone can review what Claude will do)
- Version-controlled (skill evolution is tracked in git history)
- Editable without code changes (no redeployment needed)

## The Unified Runner: `run-skill.sh`

A single script handles local and CI invocation identically:

```bash
.claude/scripts/run-skill.sh /triage 156
.claude/scripts/run-skill.sh /implement 156
.claude/scripts/run-skill.sh /audit 156
```

### What the Runner Does

1. Parses skill name and issue/PR number from args
2. Loads the SKILL.md file from `.claude/skills/<name>/`
3. Determines tool allowlist (READ_TOOLS vs WRITE_TOOLS per skill)
4. Sets `--max-turns` per skill (ask: 10, triage: 20, audit: 30, implement: 40)
5. For **read-only skills**: runs claude with `--print` flag, captures stdout, posts it as a GitHub comment via `gh issue comment`
6. For **write skills**: runs claude normally (Claude has `gh issue comment` access directly)

### The --print Architecture (Critical Bug Fix)

Early versions gave all skills access to `gh issue comment`. This caused a critical bug: triage/audit would accidentally post raw file contents as issue comments (e.g., the full `model_app.go` file appearing as a comment).

Root cause: when Claude had `gh issue comment` access during read operations, it would call it at unexpected times — sometimes with tool output rather than final analysis.

Fix: read-only skills run with `--print` (which outputs final response to stdout and exits). The shell script captures stdout and posts it as the comment. Claude never calls `gh issue comment` directly in read-only mode.

Write skills (`/implement`) still have `gh issue comment` access because they need to post the implementation log mid-run (before the PR exists).

## Tool Allowlists Per Skill

| Skill | Tools |
|-------|-------|
| `/ask` | Read, Glob, Grep, Bash(go list), Bash(gh issue/pr view) |
| `/triage` | Read, Glob, Grep, Bash(go list), Bash(gh:*) [read-only GH] |
| `/audit` | Read, Glob, Grep, Bash(go list), Bash(gh:*) [read-only GH] |
| `/implement` | Read, Write, Edit, Glob, Grep, Bash(go:*), Bash(git:*), Bash(gh:*) |
| `/create-pr` | Read, Write, Edit, Glob, Grep, Bash(go:*), Bash(git:*), Bash(gh:*) |

Explicitly excluded from read-only skills: `git`, file writes, `gh issue comment`, `gh pr create`.

The GitHub Actions workflow listens for `@claude /skill` comments on issues and PRs:

`GITHUB_TOKEN` doesn't have sufficient permissions to push branches and create PRs. A Personal Access Token (`CLAUDE_BOT_PAT`) is required. This is a known friction point for new repo setups — document it early.

2b:T81f,# Auto-Registering GraphQL Resolvers via File Naming Conventions

Instead of manually importing and registering every resolver, use a codegen script that scans the filesystem by naming convention and generates a single NestJS resolver module.

## Convention

- `m_*.ts` — mutation resolver
- `q_*.ts` — query resolver
- `report.*.ts` — paginated report query
- Files live in `src/resolvers/crudModels/{entity}/`

## Each resolver file exports a fixed interface

```typescript
@ArgsType()
export class QueryArgs {
  @Field(() => String)
  someField: string;
}

export const OutputType = SomeObjectType;

export const fn = async ({
  context,
  args,
}: {
  context: IContext;
  args: QueryArgs;
}): Promise<typeof OutputType> => {
  await canAccess(context, [perms.resource.action]);
  // ...
};
```

## generateResolvers script

Uses `glob` to find all `[mq]_*.ts` files, extracts resolver name from filename (strips `m_`/`q_` prefix), infers GQL type from prefix, then renders a Handlebars template:

```typescript
const getCrudModelResolvers = async (rootDir: string) => {
  const tsFiles = await glob(`${rootDir}/**/[mq]_*.ts`);
  return tsFiles.map((filePath) => {
    const file = path.parse(filePath);
    const gqlType = file.name[0] === 'q' ? 'Query' : 'Mutation';
    return {
      filePath: getFullPathWithoutExtension(filePath),
      resolverName: file.name.slice(2),
      gqlType,
    };
  });
};
```

## Generated module (Handlebars template)

```handlebars
@Resolver()
class CrudModelsResolver {
  constructor(readonly dbService: DbService) {}

{{#each resolvers}}
  @{{gqlType}}(() => {{resolverName}}.OutputType)
  async {{resolverName}}(
    @Args() args: {{resolverName}}.QueryArgs,
    @Context() context: IContext,
  ) {
    return await {{resolverName}}.fn({ context, args });
  }
{{/each}}
}
```

## Benefits

- Adding a new resolver = create one file, run `pnpm generateResolvers`
- No manual imports or module wiring
- Enforces consistent resolver interface across the codebase
- Easy to see all mutations/queries at a glance from the filesystem
