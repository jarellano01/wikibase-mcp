15:T81f,# Auto-Registering GraphQL Resolvers via File Naming Conventions

Instead of manually importing and registering every resolver, use a codegen script that scans the filesystem by naming convention and generates a single NestJS resolver module.

## Convention

- `m_*.ts` — mutation resolver
- `q_*.ts` — query resolver
- `report.*.ts` — paginated report query
- Files live in `src/resolvers/crudModels/{entity}/`

## Each resolver file exports a fixed interface

```typescript
@ArgsType()
export class QueryArgs {
  @Field(() => String)
  someField: string;
}

export const OutputType = SomeObjectType;

export const fn = async ({
  context,
  args,
}: {
  context: IContext;
  args: QueryArgs;
}): Promise<typeof OutputType> => {
  await canAccess(context, [perms.resource.action]);
  // ...
};
```

## generateResolvers script

Uses `glob` to find all `[mq]_*.ts` files, extracts resolver name from filename (strips `m_`/`q_` prefix), infers GQL type from prefix, then renders a Handlebars template:

```typescript
const getCrudModelResolvers = async (rootDir: string) => {
  const tsFiles = await glob(`${rootDir}/**/[mq]_*.ts`);
  return tsFiles.map((filePath) => {
    const file = path.parse(filePath);
    const gqlType = file.name[0] === 'q' ? 'Query' : 'Mutation';
    return {
      filePath: getFullPathWithoutExtension(filePath),
      resolverName: file.name.slice(2),
      gqlType,
    };
  });
};
```

## Generated module (Handlebars template)

```handlebars
@Resolver()
class CrudModelsResolver {
  constructor(readonly dbService: DbService) {}

{{#each resolvers}}
  @{{gqlType}}(() => {{resolverName}}.OutputType)
  async {{resolverName}}(
    @Args() args: {{resolverName}}.QueryArgs,
    @Context() context: IContext,
  ) {
    return await {{resolverName}}.fn({ context, args });
  }
{{/each}}
}
```

## Benefits

- Adding a new resolver = create one file, run `pnpm generateResolvers`
- No manual imports or module wiring
- Enforces consistent resolver interface across the codebase
- Easy to see all mutations/queries at a glance from the filesystem
16:T7c0,# URL-Synced Filter & Pagination State with nuqs

Each list page has a co-located `params.ts` that owns all filter/sort/pagination state in the URL via `nuqs`.

## File structure

```
src/app/(dashboard)/some-list/
├── params.ts       ← owns all URL state
├── filterBar.tsx   ← reads/writes params
└── page.tsx        ← composes FilterBar + table
```

## params.ts pattern

```typescript
"use client"

import { parseAsArrayOf, parseAsInteger, parseAsJson, parseAsString, useQueryStates } from "nuqs"
import { useMemo } from "react"
import { z } from "zod"

const sortSchema = z.array(z.object({ id: z.string(), desc: z.boolean() }))

export const useParams = () => {
  const [params, setParams] = useQueryStates(
    {
      page: parseAsInteger.withDefault(1),
      searchTerm: parseAsString,
      status: parseAsArrayOf(parseAsString).withDefault(["active"]),
      sort: parseAsJson(sortSchema.parse).withDefault([{ id: "createdAt", desc: true }]),
      limit: parseAsInteger.withDefault(50),
    },
    {
      urlKeys: { limit: "perPage" }, // rename URL key without changing JS name
    },
  )

  const updatedParams = useMemo(() => ({
    ...params,
    searchTerm: params.searchTerm ?? undefined, // null → undefined for optional GQL args
    pageIndex: params.page - 1,                 // convert 1-based page to 0-based index
  }), [params])

  return [updatedParams, setParams] as const
}
```

## Key ideas

- **Shareable URLs**: filters survive page refresh and can be copied/bookmarked
- **Type-safe parsers**: `parseAsInteger`, `parseAsString`, `parseAsArrayOf`, `parseAsJson` with Zod validation
- **`.withDefault()`**: guarantees non-null value so downstream code doesn't need null checks
- **`urlKeys`**: rename the URL param without changing the JS variable name
- **null → undefined transform**: nuqs uses `null` for absent values; GraphQL args expect `undefined`
- Returns `[params, setParams]` tuple following React hook convention
17:T18a1,# AI-Assisted Migration: Claude Skills Framework

## Why Skills Instead of Just Prompts

By the time the migration was in full swing (Jan 2026), the team had developed strong institutional knowledge about migration patterns: how to extract DAG structure, which naming conventions to use, how to write cutover DAGs safely, how to validate parity. This knowledge existed in people's heads and in review comments.

**Claude Code skills** codified this knowledge into structured, reusable AI prompts that any team member could invoke. Rather than explaining the entire pattern each time, engineers run `/create-dbt-migration dags/bi_user/ad_events/` and the skill encodes exactly what a senior engineer would know to do.

---

## The Skill Suite

Committed Jan 23, 2026 (DATA-4433), with additions through Feb 2026:

### `/create-dbt-migration` (orchestrator skill)
The top-level entry point for a full Phase 1 migration. Composes the two sub-skills via the Task tool (multi-agent):
1. Launches `/analyze-dag-dbt` as a subagent
2. Waits for the migration context YAML
3. Launches `/generate-models-dbt` as a subagent
4. Returns a review checklist for the human

### `/analyze-dag-dbt`
Reads a DAG directory and produces a structured migration context file at `.dbt-migration/migration_context_<dag_id>.yml`. Extracts:
- All SQL files referenced by `SnowflakeOperator` tasks
- `ExternalTaskSensor` wiring (upstream dependencies)
- `GreatExpectationsOperator` test configurations
- Schedule, SLA, owner metadata
- Target table names and schemas

### `/generate-models-dbt`
Takes the migration context and produces:
- Staging models (`stg_<source>__<table>.sql` + schema YAML)
- Intermediate/legacy models (`int__<name>.sql`)
- Source entries in `sources_to_dags.yml`
- dbt tests converted from Great Expectations expectations
- Task group YAML config for the new pipeline

Enforces all naming conventions automatically — the skill embeds the naming rules so generated code is always correct on first pass.

### `/validate-tables-dbt`
Post-generation validation:
1. `gdbt compile` to check SQL is valid
2. `gdbt run --select <models>` to materialize in dev
3. Row count and column comparison against original Airflow-managed tables
4. Posts Snowflake `query_id`s to the PR for reviewer confirmation

### `/create-cutover-dag`
Creates the zero-downtime cutover DAG that runs the old and new pipelines in parallel:
- Generates the cutover Python DAG from a template
- Adds column mapping validation
- Creates a draft PR automatically
- Posts a PR summary comment with migration context

### `/create-pr`
Creates pull requests with proper JIRA ticket integration, branch naming (`DATA-XXXX-<description>`), and PR body template with migration summary.

### `/create-dbt-migration-batch`
Runs multiple migrations in parallel using Claude's multi-agent Task tool:
```
/create-dbt-migration-batch dags/bi_user/ad_events/ dags/grindr_analytics/dim_profile/
```
Launches one agent per DAG simultaneously, then aggregates results.

### `/deprecated-bi-user-tables`
Reference skill for migrating deprecated `BI_USER` source tables to the new intermediate profile attribute models. Encodes the specific column mappings and semantic equivalences.

### `/merge-main`
Merges `main` into the current feature branch, auto-resolves conflicts in codegen files (since they're always regenerated), and runs `gdbt validate` to rebuild them cleanly.

---

## How AI Was Used to *Build* the Skills

The skills themselves were built with AI assistance — a meta pattern. The workflow:

1. A senior engineer performs a migration manually once, carefully
2. Claude Code (via Cursor) observes the pattern across the PR diff
3. The engineer describes the pattern to Claude: "that's always how cutover DAGs should look"
4. Claude drafts the skill YAML/Markdown
5. The engineer refines it on 2–3 real migration runs
6. The skill is committed to the repo and available to the whole team

The Cursor AI code review rules (`.cursor/rules/airflow-code-review.mdc`, `.cursor/rules/sql-code-review.mdc`) were built the same way — patterns extracted from actual review comments, formatted as machine-readable rules that Cursor applies automatically during review.

**Key insight:** The AI wasn't used to replace engineering judgment. It was used to *encode* engineering judgment so it could be applied consistently at scale.

---

## Multi-Agent Migration Workflow

The most powerful pattern: running migrations in parallel across multiple DAGs simultaneously. Example from the advertising cluster migration (DATA-4531):

```
/create-dbt-migration-batch \
  dags/bi_user/ad_data_model/ \
  dags/bi_user/ad_events/ \
  dags/bi_user/ad_revenue_rolling_avg/
```

This launches three independent agents. Each one:
- Reads the DAG
- Generates models
- Creates task group YAMLs
- Writes to non-overlapping paths in the repo

The main agent merges results and creates one PR. Work that would take 3–4 days manually was completed in a single afternoon.

---

## Cursor vs. Claude Code: Complementary Roles

Both tools were used, with distinct roles:

**Cursor (via `.cursor/rules/`):**
- In-editor AI review during development
- Applies code review findings automatically with search_replace
- Always-on linting and standard enforcement
- Used for exploratory SQL writing and model iteration

**Claude Code (via `.claude/skills/`):**
- Structured, multi-step workflow automation
- Cross-file operations (read DAG → write multiple dbt files → update YAML configs)
- PR creation and git operations
- The "do the whole thing" command

The combination meant engineers could write a model interactively in Cursor, then hand off the migration scaffolding to Claude Code skills.

---

## Metrics Impact

Looking at PR volume in the commit history:
- Pre-skills era (Nov–Jan 2025/2026): ~2–3 migration PRs per week, each taking 2–4 days
- Post-skills era (Feb–Mar 2026): 5–8 migration PRs per week, many merged same day
- Batch migrations enabled entire domain clusters to move in a single PR

The commit message pattern shifted too — from detailed implementation explanations to structured `DATA-XXXX: <verb> <thing>` messages, reflecting that engineers were spending less time on implementation details and more on review and validation.
18:Te6a,# Building a CLI + MCP Server in a TypeScript Monorepo

## Architecture
- Turborepo monorepo with pnpm workspaces
- `packages/db` — shared Drizzle ORM schema, queries, config
- `apps/cli` — Commander.js CLI (`wiki` binary)
- `apps/mcp` — local MCP stdio server (`wiki-mcp` binary)
- `apps/web` — Next.js read-only dashboard

## MCP Server is Just a Local Script
An MCP server does NOT need to be hosted. Claude Desktop and Claude Code spawn it as a local subprocess via stdio. The config just points to a binary:
```json
{ "command": "/absolute/path/to/wiki-mcp" }
```
Always use absolute paths — Claude spawns the process in a clean environment where PATH may not include your shell's bin dirs.

## Registering with Claude Code vs Claude Desktop
- Claude Desktop: `~/Library/Application Support/Claude/claude_desktop_config.json`
- Claude Code: `claude mcp add <name> <absolute-path> --scope user` (writes to `~/.claude.json`)
- These are separate configs — you need to register in both.

## tsup Bundling Gotchas
- All source files get bundled into one `dist/index.js`
- `import.meta.dirname` in the bundle = the `dist/` directory, NOT the original source file's directory
- Path calculations using `import.meta.dirname` must account for this:
  - From `dist/index.js`, `../migrations` → `apps/cli/migrations/` ✓
  - `../../migrations` from `dist/` goes two levels up to `apps/` — wrong
- Mark all runtime deps as `--external` in tsup or they won't resolve at runtime

## pnpm Global Linking
```bash
# Must be run from the package directory
cd apps/cli && pnpm link --global
cd apps/mcp && pnpm link --global
```
- Requires `PNPM_HOME` in PATH (run `pnpm setup` first)
- Creates a shell wrapper script, not a symlink
- `import.meta.dirname` resolves to the actual file path, not the wrapper location

## Workspace Package Resolution in Next.js
- `packages/db` exports TypeScript source directly via `"exports": { ".": "./src/index.ts" }`
- Next.js webpack cannot resolve `.js` imports pointing to `.ts` files
- Fix: use `transpilePackages: ["@ai-wiki/db"]` + webpack `extensionAlias`:
```ts
webpack: (config) => {
  config.resolve.extensionAlias = { '.js': ['.ts', '.tsx', '.js'] };
  return config;
}
```

## drizzle-kit Config Gotcha
- `drizzle.config.ts` is loaded by drizzle-kit's own bundler
- Cannot import from workspace TypeScript packages (no compiled output available)
- Inline any config reading logic directly in `drizzle.config.ts` using Node built-ins only
- `drizzle-kit generate` does NOT need a DB connection — only `migrate` does
- Use a try/catch around `getDatabaseUrl()` so generate works without config

## Programmatic Migrations in a CLI
Don't shell out to `pnpm db:migrate` from a setup command — it breaks when:
1. The working directory is wrong
2. The package is installed globally (no monorepo context)

Instead, use drizzle-orm's migrator directly:
```ts
import { migrate } from "drizzle-orm/postgres-js/migrator";
await migrate(db, { migrationsFolder: join(import.meta.dirname, "../migrations") });
```
Output migration files to `apps/cli/migrations/` so they're co-located with the binary.

## Postgres Connection Pool Hangs Process
The `postgres` driver keeps a connection pool open. CLI commands never exit without:
```ts
program.parseAsync().then(() => process.exit(0));
```
Exception: long-running commands like `serve` should not force-exit.

## Config File Pattern for Global CLIs
Store user config at `~/.config/<app>/config.json`, not `.env` in the project dir.
- Works regardless of where the CLI is run from
- Shared between CLI and MCP server automatically
- Still support `DATABASE_URL` env var as override (useful for CI)
19:Tf7c,# dbt Project Structure: Analytics vs AI/ML

## Two Projects, Two Philosophies

### dbt_analytics — Traditional Layered Architecture
Follows the standard dbt staging → intermediate → mart pattern:

```
dbt_analytics/models/
├── staging/           # 1:1 with raw sources, light cleaning only
│   ├── aurora/        # stg_aurora__<table>
│   ├── bi_user/       # stg_bi_user__<table>
│   ├── eventbus/      # stg_eventbus__<table>
│   └── ...
├── intermediate/      # Business logic, domain-grouped
│   ├── profile_engagement/
│   ├── profile_attributes/
│   ├── chat_message/
│   ├── product/
│   └── ...
├── mart/              # Final consumer-facing tables
│   ├── dim_<entity>   # Descriptive attributes
│   └── fact_<event>   # Events, transactions, metrics
└── reports/           # BI-ready aggregations (Looker, etc.)
```

**Strict naming conventions:**
- Staging: `stg_<source>__<table>` (double underscore after source)
- Intermediate: `int__<descriptive_name>` (double underscore prefix, NO source name)
- Mart: `dim_*` for dimensions, `fact_*` for events/transactions (never `mart_*`)
- Reports: domain-specific, often mirrors existing BI tool expectations

### dbt_aiml — Use-Case-Based Architecture
Rather than layers, organized around ML use cases:

```
dbt_aiml/models/
├── studfinder/        # Studfinder recommendation features
├── cascade/           # Cascade click/impression features
├── boostud/           # BoostuD features
├── profile_insights/  # Profile-level analytics for ML
└── ...
```

Each use case deploys to its own Snowflake schema (e.g., `AI_ML.STUDFINDER`). The use-case folder is the primary organizing concept — not whether a model is staging vs. intermediate.

---

## Incremental Patterns

The `delete+insert` strategy became the standard for all incremental models in `dbt_analytics`:

```sql
{{
    config(
        materialized='incremental',
        incremental_strategy='delete+insert',
        unique_key=['ds'],           -- or ['ds', 'hr'] for hourly
        on_schema_change='append_new_columns'
    )
}}
```

**Why `delete+insert` over `merge`:**
- Snowflake `MERGE` locks tables and is slow at scale
- `delete+insert` is faster and aligns with the daily partition model
- `on_schema_change='append_new_columns'` allows schema evolution without full rebuilds

**Hourly models** use `['ds', 'hr']` as the composite unique key and inject both `ds()` and `hr()` Jinja macros for partition filtering.

---

## Source Management

Sources are tracked in `sources_to_dags.yml` files within each project. This file is the contract between dbt models and the upstream Airflow DAGs that populate them:

```yaml
# Example entry in sources_to_dags.yml
- source: bi_user.client_event_daily
  dag_id: client_event_groups
  task_id: tasks_L.BI_USER.CLIENT_EVENT_DAILY
  table: BI_USER.REPORTING.CLIENT_EVENT_DAILY
  monitor_type: daily
```

The pipeline codegen reads this file to automatically generate `ExternalTaskSensor` tasks in the DAG — engineers never manually wire sensors.

---

## Validation via `gdbt validate`

Before any commit, engineers run `gdbt validate` from the dbt project directory. This:

1. Runs SQL linting (`sqlfluff`) with auto-fix
2. Validates task group YAML configs
3. Regenerates all `CODEGEN:` sections in task group YAMLs and DAG configs
4. Regenerates all Airflow DAG Python files from YAML configs
5. Produces a `pipeline_stats.yml` summary

CI runs `gdbt validate --ci` which fails if any files were modified — ensuring local validation was run before pushing.

---

## The `nv/` Folder Convention

Models that haven't yet been fully validated in production are placed in `models/nv/` (non-validated). This allows new models to be deployed without blocking the merge of surrounding migrations. Once validated, they're promoted out of `nv/`.
