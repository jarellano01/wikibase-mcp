26:T19f1,# HotLap: How AI Is Used and What Was Learned

## Two Distinct AI Roles

### 1. Claude as the Coaching Engine (product feature)
Claude reads structured telemetry data and writes driving coaching briefs. This is an end-user facing feature.

### 2. Claude Code as the Dev Tool (developer tool)
The entire app was built by a single developer using Claude Code, with skills defined in `.claude/skills/`. The coaching skill, the progressive guide system, the metric decomposition — all of it was designed and implemented through AI-assisted development.

---

## How the Coaching Pipeline Works

### Data Flow
```
telemetry (.db SQLite) 
  → ingest_capture.py (Python) 
  → session_analysis.py (per-lap corner scores) 
  → progressive_guide.py (aggregated reference values per corner) 
  → gather-coaching-context (collects everything into one JSON blob) 
  → Claude (writes coaching brief) 
  → saved to DB + shown in web UI
```

### What Claude Receives
The coaching context JSON includes (~12K chars after slimming from 70K):
- Session metadata (car, track, date, num_drivers, session types)
- Per-lap corner analysis: trail_brake_score, line_quality_score, min_speed_kph, gear, coasting_time, speed_ratio
- Per-component breakdowns: brake_taper, connection, rotation_readiness / lateral_g_shape, steering_smoothness, steering_corrections
- Progressive guide reference values (what good looks like for this driver at this track)
- Prior coaching at same track (most recent session)
- Prior coaching cross-track (if different track, for transfer tests)
- Fundamentals (car-level scores, focus areas, history)
- Technique wiki (relevant markdown files)

### Model Choice
- Production: `claude-sonnet-4-6` (v2 "open coach" style)
- Tested: `claude-haiku-4-5` for cost reduction — fast but coaching quality dropped noticeably on complex sessions
- v1 was a "prescriptive" style (more rigid, step-by-step instructions); v2 is an "open coach" (more contextual, adapts to what the driver actually needs)

---

## Key Learnings

### 1. Aggregates are useless for coaching — decompose everything
The first version used `trail_braked_pct` (binary: did the driver coast for less than 0.15s?). The driver repeatedly "failed" trail braking without understanding why. The fix: decompose into 3 sub-components (brake_taper, connection, rotation_readiness), score each 0-100, surface the weakest one. The driver immediately understood: "my taper is 74 but connection is 26 — I'm lifting the brake before touching the throttle."

**Pattern to reuse:** Any metric where the coaching says "improve X" but the driver doesn't know how → decompose into sub-skills, score independently, color-code.

### 2. Token reduction requires curation, not summarization
Initial coaching context was 70K+ chars (raw telemetry dumps). Slimmed to ~12K by:
- Only sending aggregated corner metrics, not raw telemetry arrays
- Limiting prior coaching to the most recent session (not the full history)
- Pre-computing all derived metrics in Python, not leaving calculation to the LLM

**Don't ask Claude to compute from raw data.** Pre-aggregate in Python. Claude's job is synthesis and communication, not math.

### 3. Prompt versioning matters when you change philosophy
Added a `coaching_version` field to distinguish v1 (prescriptive) vs v2 (open coach) prompts. When reviewing progress across sessions, the prior coaching style affects how you interpret what advice was given. Tag your prompts.

### 4. The coaching *chain* is as important as per-session analysis
Single-session coaching is shallow. The most valuable coaching pattern is the **transfer test**: when the driver moves to a new track, compare their fundamentals (coasting %, brake smoothness, steering corrections) against the last session at the previous track. Did the skill transfer or did it reset?

Cross-track coaching also enables catch-up: "Last week at Summit Point you were working on brake-throttle connection. Your first session at Oschersleben shows connection scores of 44 — better than your Summit baseline of 26. That transferred."

### 5. Horizontal coaching before vertical — AI needs explicit instructions for this
Without being explicitly told, Claude will drill into the worst corner in detail. But for beginners, this is wrong: if 3 other corners have worse consistency issues, fixing corner 8's overlap first is wasted effort. The coaching skill prompt explicitly instructs "scan all corners first, get everything to adequate, then drill." This is not intuitive for LLMs without the instruction.

### 6. Mental bandwidth model: why skill gates exist
The prompt explains to Claude *why* horizontal-first coaching works: a beginner's brain is fully occupied with vision, racing line, and braking fundamentals. There is literally no spare mental bandwidth to also process trail braking technique. The coaching should tell the driver: "Make [X] automatic before adding [Y] — not because you're not allowed to, but because your brain can't process both yet." This framing gets better driver buy-in than "you're not at that skill level yet."

### 7. Live overlay coaching requires different feedback than post-session coaching
The sim capture tool shows per-corner feedback in real-time (trail brake score, speed delta vs guide, weakest component). This feedback must be:
- **Immediate** (shown within 2-3 seconds of corner exit)
- **Single metric** (the driver can't read a paragraph mid-session)
- **Actionable in the next lap** (not a historical trend)

Post-session Claude coaching is the opposite: narrative, trend-based, multi-lap patterns.

### 8. Debug logging enables AI-to-AI comparison
The `live_log` table captures exactly what the driver saw during the session (banners, corner scores, next-corner previews). This means you can compare live feedback (what Claude's coaching said to focus on) against post-session offline analysis (what actually happened). Discovered one case where live scores were systematically lower than offline scores due to smoothing kernel difference — only visible because both sides were logged.

---

## What Didn't Work

- **Static track guides:** Required pre-built reference data per car/track combo. Never scaled. Progressive guide replaced them entirely.
- **`race_events` table:** Awkward date-based grouping. Uploads are a better container. Slated for removal.
- **Letting Python scripts mutate the `uploads` table:** Node owns that table now. Mixing ownership caused status-update races.
- **Token-heavy contexts:** 70K chars caused slow responses and increased cost. Aggressive pre-aggregation in Python is required.
27:T3579,# HotLap: Complete Metric Logic — How Every Score Is Computed

Source files: `ml/src/iracing/process_session.py` (per-lap analysis), `ml/src/iracing/queries/session_analysis.py` (session aggregation), `ml/src/iracing/queries/progressive_guide.py` (multi-session learning).

---

## Raw Telemetry Inputs

iRacing captures at 60 Hz. Each lap row in `sim_racing.telemetry` has:
- `throttle` (0-1), `brake` (0-1), `steering_angle` (degrees), `speed` (kph)
- `lap_dist` (meters from start/finish), `gear` (int)
- `lat_accel`, `long_accel` (m/s²) — needed for lateral G shape scoring

`dt = 1/60 ≈ 0.0167s` per frame. All time-based thresholds are frame-count × dt.

---

## Step 1: Corner Detection

### Braking Zone Detection (`find_braking_zones`)
A braking zone is any contiguous run where `brake > 0.1` for more than 5 frames (~83ms). Returns `(start_idx, end_idx)` pairs.

### Chicane / S-Curve Splitting (`_split_zones_by_steering`)
For each braking zone, check if steering crosses zero with significant magnitude on both sides (both > 15°). If so, the zone is a chicane — split it at the zero-crossing point. Both sub-zones must be > 5 frames to be kept as separate corners.

**Why:** A chicane produces one brake application but two distinct driving actions. Scoring them together would average out and hide technique on each direction.

### Liftoff Corner Detection (`_find_liftoff_zones`)
High-speed corners where the driver lifts off throttle but doesn't brake. Detection criteria:
1. Throttle drops from > 0.7 to < 0.3 within 1 second
2. Entry speed > 80 kph
3. Max steering angle in zone > 10°
4. Max brake < 0.1 (not a braking corner)
5. Speed drops by > 5% from entry (actually slowing down)
6. Doesn't overlap with any existing braking zone ± ~1 second

Liftoff corners get line quality scoring but NOT trail brake scoring (there's no brake to taper).

### Corner Filtering (false positive removal)
After detection, each zone is filtered before being added to results:
- **Spin/stop filter:** `min_speed < 20 kph` → skip (car nearly stopped)
- **Brake tap filter:** `min_speed > entry_speed × 0.9` → skip (car barely slowed, not a real corner)

---

## Step 2: Per-Corner Basic Metrics

For each detected corner zone `[zone_start, zone_end]`, a `pickup_window` is opened: `zone_end + 3s` of frames ahead (to catch throttle application).

`pickup_idx` = first frame after zone_end where throttle > 0.3.

**`coasting_time`**: sum of frames where `throttle < 0.05 AND brake < 0.05` from zone_start to pickup. Multiplied by dt → seconds.

**`brake_point_dist`**: `lap_dist[zone_start]` in meters from start/finish.

**`min_speed_kph`**: `min(speed[zone_start:zone_end])`.

**`exit_speed_kph`**: speed at `zone_end + pickup_idx`.

**`brake_attack_rate`**: time (seconds) from brake onset to reaching 90% of peak brake pressure.

**`speed_ratio`**: `min_speed / entry_speed` — raw fraction, not scored into 0-100.

---

## Step 3: Trail Brake Score (0-100 composite)

Weights: `brake_taper 35%`, `connection 35%`, `rotation_readiness 30%`

### Component A: Brake Taper (`_score_brake_taper`)
Measures smoothness of brake release — the gradual progressive trailing off of brake pressure through the corner.

**Algorithm:**
1. Find peak brake pressure index in zone
2. Find where brake drops below 5% after peak (`taper` segment)
3. Build ideal = linear ramp from `peak_val` to 0 across same length
4. RMSE of actual taper vs ideal: `rmse = sqrt(mean((taper - ideal)²))`
5. Score = `(1 - rmse / peak_val) × 100` → 100 means perfect linear ramp, 0 means instant dump
6. **Snap penalty:** if `taper_duration < 0.3s`, score × `(duration / 0.3)` — a real taper needs time

Score interpretation: 100 = perfectly smooth linear release over >0.3s. 0 = brake dumped instantly.

### Component B: Connection (`_score_connection`)
Measures quality of the brake-to-throttle handoff — the critical transition between pedals.

Combines two previously separate metrics (coasting gap + overlap quality) into one.

**Coasting penalty (60% weight):** Piecewise linear on `coasting_time`:
```
0.0s → 100
0.1s → 80
0.3s → 50
0.6s → 20
1.0s → 0
```

**Overlap quality (40% weight):**
- No overlap + coasting ≤ 0.05s → 90 (clean instant handoff, great)
- No overlap + coasting > 0.05s → 50 (full gap, coast score handles penalty)
- With overlap: scored on duration + brake lightness
  - Duration: ≤0.2s → 100; 0.2-0.4s → 100→70; >0.4s → 70→30
  - Lightness: avg brake during overlap; <10% → 100; >40% → 20 (riding the pedals)
  - `overlap_quality = 0.6 × duration_quality + 0.4 × lightness`

**Final: `0.6 × coast_score + 0.4 × overlap_quality`**

Score interpretation: 100 = zero coasting + brief light overlap. The sweet spot is zero dead pedal with ~0.1s of light brake as throttle comes in.

### Component C: Rotation Readiness (`_score_rotation_readiness`)
Measures whether the car is rotated (pointed toward corner exit) before the driver applies throttle. Penalizes applying gas while still turning in.

**Sub-score A: Steering unwind at throttle onset (70% of component)**

`thr_idx` = first frame after zone_end where throttle > 10%.

`unwind_frac = 1 - (steering_at_thr_idx / peak_steering)` — what fraction of the peak steering angle has been unwound when throttle comes on. 0 = still at peak (no rotation), 1 = fully straight.

Scored via piecewise breakpoints — different for heavy vs light corners:
- **Heavy corner (peak steering > 30°):**
  ```
  0.0 → 0
  0.15 → 30
  0.30 → 60
  0.50 → 85
  0.70 → 100
  ```
- **Light corner (peak steering ≤ 30°, kinks/sweepers):**
  ```
  0.0 → 30   (leniency — light corners need less unwind)
  0.10 → 60
  0.20 → 85
  0.35 → 100
  ```

Steering derivative bonus/penalty: if steering is still increasing at throttle onset → -15; if unwinding → +10.

**Sub-score B: Throttle reapplications (30% of component)**
Counts throttle lift-reapply events (>10% → <5% → >10%) after brake release:
```
0 reapplications → 100
1 → 55
2 → 20
3+ → 0
```

**Final: `0.70 × unwind_score + 0.30 × reapp_score`**

---

## Step 4: Line Quality Score (0-100 composite)

Weights: `steering_corrections 35%`, `steering_smoothness 35%`, `lateral_g_shape 30%`

Window: expanded to `zone_start` through `pickup_end + 1s` (covers full corner including exit).

### Component A: Steering Corrections
Counts mid-corner steering reversals — real direction changes indicating the driver is fighting the car.

**Algorithm:**
1. Low-pass filter steering with 200ms moving average (~12 frames at 60Hz) to remove force feedback noise
2. Compute filtered steering rate (derivative)
3. Threshold: `max(8, min(30, 2000/avg_speed))` degrees/frame — higher speed = slightly more tolerance
4. Count zero-crossings in rate signal (sign changes above threshold)

Piecewise scoring:
```
0 → 100
1 → 90
3 → 70
5 → 50
7 → 30
10 → 10
14 → 0
```

### Component B: Steering Smoothness
Measures jerkiness of steering inputs, normalized by what's physically "normal" for this corner.

`steer_rate_std = std(derivative of filtered steering)`

Three normalization factors:
- **Speed factor:** `(avg_speed / 100)^0.5` — sqrt scaling, faster corners naturally have more rate
- **Range factor:** `1 + min(steer_range / 80, 0.5)` — wider steering arc tolerates more rate (cap at 1.5x)
- **Brake factor:** `1 + avg_brake × 0.3` — heavy braking unsettles the car (cap at ~1.24x)

`normalized_std = steer_rate_std × speed_factor / (range_factor × brake_factor)`

Piecewise scoring:
```
0 → 100
25 → 95
45 → 80
65 → 60
85 → 40
110 → 20
140 → 0
```

### Component C: Lateral G Shape
A committed, correct line produces a single smooth peak in lateral acceleration. Multiple peaks indicate the driver adjusted mid-corner (changed line, corrected).

**Algorithm:**
1. Light smoothing of lateral G (kernel size = `max(3, min(7, window_length/20))`)
2. Count distinct peaks: regions where lateral G exceeds 40% of max G
3. Count complete above-threshold cycles (below→above→below transitions)
4. Score: 1 peak → 100, 2 → 60, 3 → 20, 4+ → 0

Falls back to 50 if not enough G force to judge (< 0.1 G max), or if lat_accel data unavailable.

---

## Step 5: Session-Level Aggregation (`query_session_analysis`)

### Warm-Up Exclusion
First 2 laps per sub-session tagged as `is_warmup = True`. Cold tires + driver not yet in rhythm. All aggregate statistics are computed from "flying laps" only (non-warmup). If < 2 flying laps exist, fall back to all laps.

### Consistency Metrics
From flying laps:
- `lap_time_std`, `lap_time_mean`, `lap_time_best`, `lap_time_worst`
- **Theoretical best:** min sector time per sector summed across flying laps
- **Sector variance:** std per sector across flying laps
- **Brake point variance:** std of brake point distances per corner cluster (in meters)
- **Min speed variance:** std of corner min speeds per cluster

### Corner Clustering (`_cluster_corners`)
Groups corner observations from all flying laps by distance. Two observations cluster together if their brake point distances are within 80m. Used for per-corner consistency stats across laps within a single session.

### Orphan Cluster Merging (session-level)
After matching clusters to named guide corners, any unmatched ("orphan") cluster within 150m of a matched cluster gets merged into it. Prevents split braking zones from appearing as phantom `C4`, `C6` extra corners.

### Best vs Average
Compares best lap vs session average on: throttle_mean, full_throttle_pct, coasting_pct, steering_smoothness, throttle_smoothness.

### Race Analysis (when session_type = "race")
Clean air vs traffic technique comparison: separate averages of coasting_pct and steering_smoothness for laps tagged as `clean` vs `traffic` or `heavy_traffic`. Traffic proximity from `proximity.close_ahead_pct` in lap_analysis.

---

## Step 6: Progressive Guide — Multi-Session Learning

The progressive guide accumulates corner data across all sessions for a car/track combo. Each corner observation is evaluated independently — a lap where the driver coasts through T1 contributes nothing to T1's reference but may contribute high-quality data for T5.

### Phase 1: Collect All Observations
Query all laps for the car/track combo. For each lap's `lap_analysis->'corners'`, extract: `dist_m` (brake_point_dist), `min_speed_kph`, `trail_brake_score`, `coasting_time`, `line_quality_score`, `gear_at_apex`, `lap_time`, `session_id`.

### Phase 2: Cluster by Distance
Sort all observations by distance, then sweep: each new observation either joins the current cluster (within 80m of cluster center) or starts a new one. Cluster center updates as running mean.

### Phase 3: Outlier Removal (IQR fence)
For each cluster's speed distribution:
```python
q1, q3 = percentile(speeds, [25, 75])
iqr = q3 - q1
upper_fence = q3 + 2.0 * iqr  if iqr > 5  else  q3 + 30
lower_fence = max(1, q1 - 2.0 * iqr)  if iqr > 5  else  max(1, q1 - 30)
```
Observations outside this fence are data errors (mis-clustered zones, off-track events). If IQR is very small (<5 kph), use flat ±30 kph fence instead.

### Phase 4: Effort Filter (P30 threshold)
After outlier removal, discard observations below the 30th percentile of speed for that corner. These are "coasting" laps — the driver wasn't trying hard at this corner. Reference values should only reflect genuine attempts.

### Phase 5: Reference Speed (CV-gated)
`cv = std(high_effort_speeds) / mean(high_effort_speeds)` — coefficient of variation.

| CV | Interpretation | Reference Speed |
|---|---|---|
| < 5% | Settled technique — consistent | P80 of high-effort speeds |
| < 10% | Developing | P70 |
| ≥ 10% | Experimenting | Median (conservative) |

**Why percentiles, not max:** The driver may occasionally fluke a fast corner entry that they can't reproduce. The guide tracks what they can *consistently* achieve.

### Phase 6: Confidence Rating
```
high:   CV < 5%  AND unique_sessions ≥ 3  AND high_effort_count ≥ 15
medium: CV < 10% AND unique_sessions ≥ 2  AND high_effort_count ≥ 5
low:    everything else
```

Three confidence gates also map to the coaching "skill stage" cold-start: learning (<3 sessions), developing (3-5), stable (5+).

### Phase 7: Overdriving Detection
For each corner in the guide:
```
overdriving = (P50_speed > ref_min_speed) AND (tb_mean < 50 OR lq_mean < 50)
```
The driver is going faster than their reference speed, but their technique scores are degraded — they're beyond their limit.

### Corner Naming (Lovely Track Data)
Two-step fuzzy match:
1. Find the closest matching track in `sim_racing.lovely_tracks` by name similarity (pg_trgm, threshold > 0.3), then pick closest by track length (using actual telemetry max `lap_dist` as length)
2. For each cluster, find which Lovely turn's distance range it falls in → assign the human name ("Esses", "Horseshoe", etc.)

## The Skill Suite

Committed Jan 23, 2026 (DATA-4433), with additions through Feb 2026:

### `/create-dbt-migration` (orchestrator skill)
The top-level entry point for a full Phase 1 migration. Composes the two sub-skills via the Task tool (multi-agent):
1. Launches `/analyze-dag-dbt` as a subagent
2. Waits for the migration context YAML
3. Launches `/generate-models-dbt` as a subagent
4. Returns a review checklist for the human

- Schedule, SLA, owner metadata
- Target table names and schemas

1. `gdbt compile` to check SQL is valid

Creates the zero-downtime cutover DAG that runs the old and new pipelines in parallel:
- Generates the cutover Python DAG from a template
- Adds column mapping validation
- Creates a draft PR automatically
- Posts a PR summary comment with migration context

### `/create-pr`
Creates pull requests with proper JIRA ticket integration, branch naming (`DATA-XXXX-<description>`), and PR body template with migration summary.

### `/create-dbt-migration-batch`
Runs multiple migrations in parallel using Claude's multi-agent Task tool:

Launches one agent per DAG simultaneously, then aggregates results.

Reference skill for migrating deprecated `BI_USER` source tables to the new intermediate profile attribute models. Encodes the specific column mappings and semantic equivalences.

### `/merge-main`

## How AI Was Used to *Build* the Skills

The skills themselves were built with AI assistance — a meta pattern. The workflow:

1. A senior engineer performs a migration manually once, carefully
2. Claude Code (via Cursor) observes the pattern across the PR diff
3. The engineer describes the pattern to Claude: "that's always how cutover DAGs should look"
4. Claude drafts the skill YAML/Markdown
5. The engineer refines it on 2–3 real migration runs
6. The skill is committed to the repo and available to the whole team

The Cursor AI code review rules (`.cursor/rules/airflow-code-review.mdc`, `.cursor/rules/sql-code-review.mdc`) were built the same way — patterns extracted from actual review comments, formatted as machine-readable rules that Cursor applies automatically during review.

**Key insight:** The AI wasn't used to replace engineering judgment. It was used to *encode* engineering judgment so it could be applied consistently at scale.

## Background

The data platform originally ran entirely on Airflow DAGs: SQL files living next to Python orchestration files, table creation handled imperatively via `SnowflakeOperator` tasks, and pipeline dependencies wired manually through `ExternalTaskSensor` chains. Over time this created:

- **Maintenance drag** — SQL changes required touching both the query file and the DAG wiring
- **No lineage** — no automatic tracking of which tables depended on which sources
- **Inconsistent testing** — Great Expectations tests were bolted on per-DAG, not part of the model contract
- **Hard to onboard** — new engineers had to understand both Airflow and bespoke SQL patterns to contribute

The migration to dbt solved all of these by decoupling *transformation logic* (dbt) from *orchestration* (Airflow), while building a codegen layer so users could configure pipelines entirely from YAML in the dbt repo.

## What It Is

Cronos uses an AI-assisted development workflow where all work flows through GitHub issues and pull requests. Claude is available at each stage via "skills" — structured prompts invoked by commenting on issues or PRs.

## The Four Stages

```
Create Issue → Triage/Audit → Implement → Review → Merge → Deploy
```

### 1. Create Issue
All work starts as a GitHub issue. No fixed template required — describe the problem clearly enough for triage to work. Issues can be created manually or via `@issue-creator`.

### 2. Triage or Audit
Before coding, analyze the issue:

- **`@claude /triage`** — For simple or well-understood issues. Produces: affected files, proposed approach, testing strategy, readiness signal (🟢/🟡/🔴).
- **`@claude /audit`** — For complex or ambiguous issues. Deeper exploratory analysis: trade-offs, open questions, phased implementation plan. May suggest splitting the issue.

Both skills are conversational — they read ALL prior comments, so the thread accumulates shared context. Iterate until the approach feels right.

### 3. Implement
Once the approach is settled:
```
@claude /implement
```
Claude reads the issue, all comments (including triage/audit decisions), writes the code, runs tests, posts an implementation log, commits to a feature branch, and opens a draft PR.

### 4. Review
On any PR:
```
@claude /review
```
Senior-level review across four dimensions: security, performance, simplification, and testing. Fixes bugs within scope; flags issues outside scope.

## Invocation Methods

Skills work identically locally and in CI:

```bash
# Locally
.claude/scripts/run-skill.sh /triage 156
.claude/scripts/run-skill.sh /implement 156

# In GitHub (via issue comment)
@claude /triage
@claude /implement
@claude /audit
@claude /ask What does ApproveInvoice do?
```

## Human-in-the-Loop Design

This is explicitly **not** autonomous. The workflow design:
- Claude proposes and analyzes; **humans decide**
- The readiness signal (🟢/🟡/🔴) forces a go/no-go decision before implementation
- `/triage` is manually invoked — not automatic on issue creation
- PRs always start as **drafts** — humans mark them ready

## Key Principle: Comments Are the Workflow

GitHub issue comments drive everything. No separate tickets, no status database — the conversation IS the record. Every skill re-reads all prior comments to stay current.

## Label System

Issues move through stage labels:
- `stage:backlog` → `stage:triaged` → `stage:audited` → `stage:in-progress`

Type labels: `type:bug`, `type:feature`, `type:enhancement`, `type:refactor`
Priority labels: `priority:critical`, `priority:high`, `priority:medium`, `priority:low`
