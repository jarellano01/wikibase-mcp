22:T13cf,# AI Workflow Evolution: How Cronos Built Its Agentic Dev System

## Timeline Overview

The entire workflow was built in approximately 10 days in March 2026, going from zero automation to a polished skill-based system.

## Phase 1: Foundation (through PR #118, ~March 11)

**What was built:**
- Initial `claude.yml` GitHub Actions workflow
- Basic `/triage` skill
- `feature-discovery` skill (local-only, for complex pre-work research)
- Strategic foundation in issue #129: a detailed analysis of the codebase's agentic development readiness, identifying 5 priority levels of work

**Issue #129 specifically identified:**
- Commit pattern analysis (contributor concentration, PR size patterns)
- CI gaps (missing test coverage, no staging environment)
- Code structure readiness for AI-assisted modification
- 10+ child issues to build out the workflow

PR #126 added the architecture and business overview docs (`docs/architecture.md`, `docs/business-overview.md`) that now serve as mandatory context loaded by every skill.

## Phase 2: Rapid Infrastructure (March 12–17, PRs #143–158)

This phase was almost entirely fixing bugs discovered through real usage:

| PR | Fix |
|----|-----|
| #143 | Skills routing — parsing `@claude /triage` from comment body |
| #146 | Max-turns too low (15) — audit ran out of turns mid-analysis; raised to 30 |
| #150 | Comment content bug — Claude posting raw file contents instead of analysis |
| #151 | Token permissions — `workflows:write` is invalid, removed |
| #158 | Unified skill runner (`run-skill.sh`) replacing per-skill workflow files |
| #159 | `/audit` skill added |
| #160 | Readiness signals (🟢/🟡/🔴) added to triage; phased plan template added to audit |
| #173 | `/create-pr` skill added |
| #175 | Skill name parsing bug fixed (stripping `@claude /` prefix) |
| #178 | PR creation skipped — implement was completing after commit step; added explicit "ALWAYS attempt gh pr create" |
| #180 | `CLAUDE_BOT_PAT` documented — standard `GITHUB_TOKEN` insufficient for branch push + PR create |
| #181 | `workflows:write` permission removed (invalid, caused workflow failures) |

**The comment content bug (#150) was the most impactful fix:**
Claude had `gh issue comment` access during triage/audit. It would call it at unexpected times with raw file output rather than final analysis. Fix: read-only skills run with `--print` — the shell captures stdout and posts it. Claude never directly calls `gh issue comment` in read-only mode.

## Phase 3: Refinement Through Real Usage (March 18+, PRs #159–188)

With infrastructure stable, the workflow was used on real issues:

- **Issue #127** (N+1 invoice timeout, 82s for 231-entry invoice): Went through audit → produced root cause analysis → implemented fix → added `dbQueryCounter` test pattern → added to CLAUDE.md
- **Issue #136** (test coverage gaps): Multi-round audit conversation. Team asked follow-up questions, got refined answers. The conversational loop materially changed the implementation plan.
- **Issue #171** (staff page slow): First time a second team member used the workflow end-to-end: `@claude /triage` → comment with selected option → `@claude /implement let's implement Option A`
- **Issue #176** (monetary type consistency): `/audit` surfaced two bugs not in the original issue scope
- **Issue #185** (billing edge case): Triage returned 🟡 Yellow with a question → team answered in comment → `/implement` read the answer and proceeded

## Key Lessons

### What worked immediately
1. **Readiness signals** stopped premature implementation starts
2. **Implementation logs** created searchable history on issues
3. **Local/CI parity** meant developers could test skills locally before relying on CI
4. **All-comments-read** design meant the triage → implement handoff was seamless

### What needed iteration
1. **Max turns** — 15 was too low for audit; always tune per skill based on observed usage
2. **Permission architecture** — read-only skills must not have write tool access; the `--print` capture pattern is the right solution
3. **PR creation** — without explicit emphasis, Claude would complete the implement flow without creating the PR; explicit instruction in the prompt fixed it
4. **GitHub token** — `GITHUB_TOKEN` is insufficient; document `CLAUDE_BOT_PAT` requirement upfront for any repo adopting this pattern

### Design decisions that held up
- Skills as files (not code) — human-readable, version-controlled
- Unified runner — local/CI parity with zero divergence
- Human-in-the-loop — never automatic on issue creation; always manually invoked
- Draft PRs always — review gate before merge

## The Spread to Other Contributors

The workflow was initially used only by the lead developer. Within the first week of Phase 3, a second team member discovered it independently and used `@claude /triage` then `@claude /implement` on issue #171 in a single day — without any onboarding. This confirmed that the workflow was discoverable and usable by the broader team.
23:Tc9d,# Implement Skill: Implementation Log Pattern

## The Pattern

Every `/implement` run posts a structured comment on the issue before creating the PR. This "implementation log" is mandatory — it is never skipped even for trivial changes.

## Implementation Log Structure

```markdown
## Implementation Log

### What Was Implemented
[Option chosen, coverage, deviations from proposed approach]

### Repo Setup Notes
[Pre-existing test failures, required stubs, environment issues]

### Files Changed
| File | Change | Rationale |
|------|--------|-----------|
| invoice.go:142 | Modified ApproveInvoice | ... |
| invoice_test.go | Added TestApproveInvoice_N1 | N+1 guard |

### Key Decisions / Findings
[Non-obvious choices, discovered edge cases, patterns followed]

### Test Results
```
ok  cronos  0.847s
```

### Open Questions
[Anything left unresolved that the reviewer should know]
```

## Why It Matters

1. **Handoff record** — When a future session picks up the issue, this log provides context that isn't in the code diff
2. **Reviewer context** — PR reviewers can see what option was chosen and why, without re-reading the full triage thread
3. **Searchable history** — The implementation log is on the issue, which stays searchable even after the PR is merged
4. **Surfaces surprises** — "Key Decisions / Findings" is where unexpected bugs or constraints surface before the PR review

## The 10-Step Implement Process

1. Load project context (business docs, CLAUDE.md)
2. Read issue + ALL comments (triage decisions, audit findings, answers to blockers)
3. Verify repo setup: `go build ./...`, `go test . -count=1`, document pre-existing failures
4. Identify blockers (missing AutoMigrate, external service calls, accounting implications)
5. Implement (follow existing patterns exactly; match fixture chains from closest test)
6. Run tests: targeted first, then full suite for regressions
7. **Post implementation log**
8. Commit to feature branch (`feat/issue-N-short` or `fix/issue-N-short`)
9. **ALWAYS create draft PR** (never skip — this was a recurring bug in early versions)
10. Post PR link on the issue

## The PR Creation Bug (Historical)

In early versions of the skill, the implement flow would complete after step 8 (commit) and exit before creating the PR. This happened because the skill prompt didn't explicitly emphasize step 9. The fix: add "ALWAYS attempt `gh pr create` — do not skip this step even if you think you're done" with prominent formatting in the skill prompt. The skill now runs with `--max-turns 40` to ensure there's headroom to reach step 9.

## GORM Safety Rules (From CLAUDE.md)

The implement skill enforces these rules to prevent N+1 regressions:

```go
// WRONG — triggers BeforeSave on all associated entries
a.DB.Model(&invoice).Updates(map[string]interface{}{"state": "paid"})

// RIGHT — empty struct + explicit WHERE
a.DB.Model(&Invoice{}).Where("id = ?", invoice.ID).Updates(map[string]interface{}{"state": "paid"})
```

And for N+1 test guards:
```go
counter := &dbQueryCounter{}
db.Use(counter)
before := counter.get()
app.SomeOperation(id)
if counter.get()-before > maxAllowed {
    t.Errorf("N+1 regression: %d queries", counter.get()-before)
}
```
24:T1138,# Skill System Architecture: File-Based Prompts with a Unified Runner

## Skills Are Files

Each skill is a `SKILL.md` file stored in `.claude/skills/<name>/SKILL.md`. The file has YAML frontmatter followed by the full prompt:

```yaml
---
name: triage
description: Analyze a GitHub issue and propose an implementation approach
allowed-tools: Read, Glob, Grep, Bash(gh:*), Bash(go:list)
---

You are a senior engineer on the Cronos project...
[full prompt follows]
```

The skill file IS the prompt — loaded verbatim at runtime. This makes skills:
- Human-readable (anyone can review what Claude will do)
- Version-controlled (skill evolution is tracked in git history)
- Editable without code changes (no redeployment needed)

## The Unified Runner: `run-skill.sh`

A single script handles local and CI invocation identically:

```bash
.claude/scripts/run-skill.sh /triage 156
.claude/scripts/run-skill.sh /implement 156
.claude/scripts/run-skill.sh /audit 156
```

### What the Runner Does

1. Parses skill name and issue/PR number from args
2. Loads the SKILL.md file from `.claude/skills/<name>/`
3. Determines tool allowlist (READ_TOOLS vs WRITE_TOOLS per skill)
4. Sets `--max-turns` per skill (ask: 10, triage: 20, audit: 30, implement: 40)
5. For **read-only skills**: runs claude with `--print` flag, captures stdout, posts it as a GitHub comment via `gh issue comment`
6. For **write skills**: runs claude normally (Claude has `gh issue comment` access directly)

### The --print Architecture (Critical Bug Fix)

Early versions gave all skills access to `gh issue comment`. This caused a critical bug: triage/audit would accidentally post raw file contents as issue comments (e.g., the full `model_app.go` file appearing as a comment).

Root cause: when Claude had `gh issue comment` access during read operations, it would call it at unexpected times — sometimes with tool output rather than final analysis.

Fix: read-only skills run with `--print` (which outputs final response to stdout and exits). The shell script captures stdout and posts it as the comment. Claude never calls `gh issue comment` directly in read-only mode.

Write skills (`/implement`) still have `gh issue comment` access because they need to post the implementation log mid-run (before the PR exists).

## Philosophy

The workflow is designed so that the "easy path" is also the safe path. Rather than relying on prompts to prevent mistakes, safety is enforced in scripts that run before/after tool calls.

## Hooks (Pre/Post Tool Execution)

### `protect-main-on-pr.sh` (Pre-tool hook on `gh pr create`)

**Problem it solves:** Developers sometimes accumulate commits on `main` locally (especially when working quickly), then try to open a PR from `main` — which targets `main` as both source and base.

**What it does:**
1. Detects if the current branch is `main` with unpushed commits
2. Extracts a branch name from the most recent commit message
3. Automatically creates a new branch from those commits
4. Moves the commits to the new branch
5. Allows the `gh pr create` to proceed normally

This runs **before** every `gh pr create` call, so neither Claude nor developers can accidentally create PRs from main.

### `capture-issue-create.sh` (Post-tool hook on `gh issue create`)

**Problem it solves:** Issues are created without being reviewed, leading to incomplete or ambiguous issues that waste triage time.

**What it does:**
1. Parses the successful `gh issue create` output to extract the new issue number
2. Outputs a reminder: "Issue #N created. Don't forget to run `@issue-reviewer #N` to validate."

This creates a nudge without making review mandatory (the decision remains with the human).

## Sandboxed GitHub CLI: `gh.sh`

A wrapper around `gh` that enforces a command allowlist. Skills that only need to read use `gh.sh` instead of raw `gh`.

**Allowed operations:**
```
gh issue view
gh issue list
gh issue search
gh issue label list
gh pr view
gh pr diff
gh pr list
```

**Blocked operations:** anything that writes — `gh issue create`, `gh issue close`, `gh issue comment`, `gh pr create`, `gh pr merge`, `gh pr review`, branch operations.

The script validates both the command and its flags before passing through to `gh`.

## `edit-issue-labels.sh`

Manages issue labels safely:
```bash
edit-issue-labels.sh --issue 156 --add-label stage:triaged --remove-label stage:backlog
```

**Safety features:**
1. Fetches the valid label list from GitHub before applying
2. Validates every label in `--add-label` against the valid list
3. Reports what was added/removed (doesn't silently fail on invalid labels)

Prevents skill prompts from accidentally applying non-existent labels (which would silently do nothing in raw `gh`).

## The Pattern

Every `/triage` output ends with a readiness signal — a traffic-light assessment of whether the issue is ready to implement:

- **🟢 Green** — Clear scope, low risk, localized changes, existing tests cover the area. Implement immediately.
- **🟡 Yellow** — Moderate uncertainty, touches shared logic, edge cases unclear. Review triage output first, possibly run `/audit`.
- **🔴 Red** — Affects accounting/GL, requires migrations, state machine changes, scope too vague. Run `/audit` first; consider splitting the issue.

## Why It Works

Without a readiness signal, developers tend to start implementing before the scope is clear. The traffic light forces an explicit decision gate: "Are we confident enough to proceed?" This stopped a pattern of implementation work starting on under-scoped or too-large issues.

## Triage Output Structure

```
## Affected Areas
- Models: [list]
- Files: [list]
- State Transitions: [if any]
- Risk Areas: [if any]

## Proposed Approach
1. [Concrete bullet]
2. [Concrete bullet]
...

## Testing Strategy
[What tests to write/update]

## Related Issues
[Links if any]

## Open Questions
[Blockers or important unknowns]

## Readiness: 🟢 Green
[One-line rationale]
```

## Escalation Path

- Green → go straight to `/implement`
- Yellow → discuss in comments, possibly `/audit`, then `/implement`
- Red → always run `/audit`; if audit finds it's too large, break into smaller issues, then triage/implement each

## Max Turns

The triage skill runs with `--max-turns 20`. This is enough for most issues (scoped analysis of 3-10 files). If triage hits the limit, it's a signal the issue is too large and should be escalated to `/audit` (which runs at `--max-turns 30`).

## Contrast with Audit

Triage is **prescriptive** — it identifies the problem and proposes a specific approach. Audit is **exploratory** — it surfaces trade-offs and options without prescribing. Use triage by default; escalate to audit only when triage returns red or the issue is explicitly architectural.
