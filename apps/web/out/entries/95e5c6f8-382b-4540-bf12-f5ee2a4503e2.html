How the pipeline framework works: engineers configure DAGs entirely from YAML in the dbt repo, and gdbt codegen produces the Airflow Python files, sensors, and task groups automatically.

Pipeline Framework: YAML-to-Airflow Codegen

Core Idea

Engineers should never need to write Airflow Python to add a new dbt pipeline. They configure what runs in YAML files inside the dbt project, run gdbt validate, and the framework generates correct, production-grade Airflow DAG Python files.

This decoupling means:

  • Data engineers work entirely in the dbt repo
  • DAG Python files are treated as build artifacts, not source code
  • Sensor wiring, task groups, and scheduling are derived from the dbt model graph

Config Hierarchy

dbt_analytics/pipelines/
├── config.yaml                    # Project-level settings
├── dags/                          # One YAML per Airflow DAG
│   ├── profile_engagement.yml
│   ├── product_analytics.yml
│   └── ...
└── task_groups/                   # One YAML per logical task group
    ├── intermediate/
    │   ├── profile_engagement.yml
    │   ├── product.yml
    │   └── profile_attributes/
    │       ├── profile_device.yml
    │       └── ...
    ├── mart/
    └── staging/

DAG Config (dags/<name>.yml)

name: profile_engagement
description: Daily pipeline for profile engagement models.

output_dag: dags/dbt/analytics/profile_engagement.py
schedule: daily 18:00
start_date: 2026-03-08
catchup: true
depends_on_past: true

config:
  dbt_project: analytics
  warehouse:
    local: DE_ETL_WH
    prod: DE_ETL_WH

task_groups:
- file: intermediate/profile_engagement.yml
- file: mart/profile_engagement.yml

# CODEGEN sections below — do not edit manually

Task Group Config (task_groups/<layer>/<name>.yml)

name: profile_engagement
dbt:
  selector: path:models/intermediate/profile_engagement

# CODEGEN: Auto-populated by gdbt validate
included_models:
- key: int__profile_taps_daily
  type: incremental
  target: BI_USER.REPORTING.PROFILE_TAPS_DAILY

upstream_task_groups:
- intermediate/product

sources:
- bi_user.client_event_daily
- eventbus.login_event

Codegen Process

When gdbt validate runs, the Python codegen layer (generate_dag.py, generate_pipeline.py) does:

  1. Parse task group YAMLs — load dbt selectors, extract included models by running dbt ls
  2. Resolve upstream dependencies — for each source in a task group, look up sources_to_dags.yml to find the upstream DAG + task ID
  3. Compute execution deltas — given the downstream DAG schedule and upstream DAG schedule, calculate the timedelta for each ExternalTaskSensor
  4. Populate CODEGEN sections — write auto-discovered sensors, upstream DAGs, downstream DAGs, and stats back into the YAML files with # CODEGEN: comment markers
  5. Generate Airflow Python DAG — render dag_template.jinja2 with all config, producing a fully valid Airflow Python file
  6. Validate references — check that all referenced task group files exist, all sensor external_dag_ids exist

The # CODEGEN: comment is the contract: anything below it in a YAML or Python file is machine-written and will be overwritten on the next gdbt validate --fix.

Generated DAG Structure

The generated Python DAG always follows the same structure:

# CODEGEN: This file is code-generated and should not be modified manually
# Generated from: dbt_analytics/pipelines/dags/profile_engagement.yml
# To regenerate: cd dbt_analytics && gdbt validate

from cosmos import DbtTaskGroup, ProjectConfig, RenderConfig
from plugins.util import pipeline_builder as pb

SNOWFLAKE_WAREHOUSE = pb.utils.get_warehouse_for_env(local_warehouse='DE_ETL_WH', prod_warehouse='DE_ETL_WH')

dag_config = pb.config.DagConfig(dag_id='profile_engagement', ...)

with DAG(**dag_config.to_dag_kwargs()) as dag:
    # For each task group:
    with TaskGroup('intermediate_profile_engagement') as tg:
        with TaskGroup('source_sensors') as _sensors:
            ExternalTaskSensor(task_id='wait_for_...', ...)
        
        DbtTaskGroup(
            group_id='dbt_models',
            project_config=ProjectConfig(...),
            render_config=RenderConfig(select=['path:models/intermediate/profile_engagement'], ...),
            ...
        )

Astronomer Cosmos (DbtTaskGroup) handles the actual dbt model execution within Airflow, providing per-model task visibility in the Airflow UI.

Sensor Framework

The sensor system evolved significantly:

v1 (manual): Engineers wrote ExternalTaskSensor directly in DAG Python files

v2 (sources_to_dags.yml): Sources declared in YAML; sensors auto-generated from the file

v3 (task group sensors): Sensors scoped to the task group that needs them, not the DAG level

v4 (sensor overrides, DATA-4545): Per-task-group sensor_config in YAML allows overriding sensor type, retries, and execution delta without touching codegen:

# In a task group YAML, under sensor_config
sensor_config:
  freshness_check_override:
    source: chat_sensitive.chat_platform
    table: CHAT_SENSITIVE.REPORTING.CHAT_PLATFORM
    monitor_type: hourly
  external_task_sensor_override:
    external_dag_id: some_other_dag
    external_task_group_id: specific_task_group

Centralized source sensors (DATA-4583): A dedicated dbt_shared_source_sensors DAG performs freshness checks on all shared upstream tables. Domain DAGs wait on this DAG rather than each independently checking the same sources.

Pipeline Stats

Every gdbt validate run produces pipeline_stats.yml which is published as a PR comment, showing:

profile_engagement:
  task_group_ct: 8
  included_models_ct: 73
  sensors_ct: 29

This gives reviewers immediate visibility into the scale of what's being deployed.
