Inline Auto-Save Pattern for Form Fields

idea · react, forms, ux, apollo, patterns, client:bgc · 3/21/2026
Pattern for immediately persisting individual field changes without a submit button: fire a mutation on onChange, show isSaving state, invalidate Apollo cache, and use form.setValue with shouldDirty:false to avoid false dirty state.

Inline Auto-Save Pattern for Form Fields

For forms where each field change should persist immediately (no submit button), this pattern fires a mutation directly in the field's onChange handler.

Core implementation

const [isSaving, setIsSaving] = useState(false)
const { runMutation: updateMutation } = useUpdateEntity()

const saveField = useCallback(async (params: Partial<EntityFields>) => {
  if (!id || isSaving) return
  setIsSaving(true)
  try {
    const result = await updateMutation({ variables: { id, ...params } })
    if (result?.success) {
      // Invalidate Apollo cache for related queries
      clearRootQueries(client, ["report_EntityListPage"])
      toast.success("Saved")
      refetchData?.()
    }
  } catch (err: any) {
    toast.error(err?.message || "Failed to save")
  } finally {
    setIsSaving(false)
  }
}, [id, isSaving, updateMutation, client, refetchData])

Field usage

<InlineDatePicker
  value={data?.someDate}
  onChange={async (date) => {
    await saveField({ someDate: date })
    // Update local form state without marking it dirty
    form.setValue("someDate", date, { shouldDirty: false })
  }}
  disabled={isLocked || isSaving}
/>

Key details

  • isSaving guard: prevents concurrent saves from the same component
  • shouldDirty: false: keeps RHF from thinking the form has unsaved changes after a successful save
  • Disable inputs while saving: disabled={isLocked || isSaving} prevents race conditions
  • Cache invalidation: clearRootQueries refreshes list pages that show the same data
  • isLocked pattern: separate business logic flag (e.g. invoice is finalized) from save-in-progress flag
