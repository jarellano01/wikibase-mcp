1f:T70a,# Wiki Idea Promotion Model: Aggregate Entries Supersede Source Entries

As a wiki grows, related entries accumulate and create clutter. Rather than deleting them, promote them into aggregates that supersede the originals.

## Levels

| Level | Description | Example |
|-------|-------------|---------|
| 1 — Observation | Raw note from a specific project or context | "BGC uses nuqs for filter state" |
| 2 — Pattern | Synthesized from 2+ observations | "URL-synced filter state pattern" |
| 3 — Principle | Distilled from multiple patterns, broadly applicable | "Externalize ephemeral UI state to the URL" |

## Conventions (using tags + content links)

**Source entries (a, b, c)** — add tag `superseded` and note the aggregate in content:
```
tags: [...original tags, "superseded"]
> Superseded by: [Aggregate Entry Title] {uuid}
```

**Aggregate entry (d)** — add tag `aggregate` and list sources:
```
tags: [..., "aggregate"]
> Emerged from:
> - [Entry A title] {uuid}
> - [Entry B title] {uuid}
> - [Entry C title] {uuid}
```

## Search behavior intent

- Default searches should prefer entries **without** the `superseded` tag
- Explicitly searching `superseded` or browsing by ID still surfaces them
- This keeps the "active" knowledge surface clean without losing lineage

## Why keep superseded entries?

- **Lineage**: shows how the idea evolved and what specific cases informed it
- **Context recovery**: if the aggregate is too abstract, the originals have the concrete examples
- **Non-destructive**: avoids the loss of nuance that comes with deletion

## Promotion triggers

Consider aggregating when:
- 3+ entries share overlapping tags and similar summaries
- An entry is referenced repeatedly across projects
- A pattern has been validated across multiple clients/contexts
20:Te6a,# Building a CLI + MCP Server in a TypeScript Monorepo

## Architecture
- Turborepo monorepo with pnpm workspaces
- `packages/db` — shared Drizzle ORM schema, queries, config
- `apps/cli` — Commander.js CLI (`wiki` binary)
- `apps/mcp` — local MCP stdio server (`wiki-mcp` binary)
- `apps/web` — Next.js read-only dashboard

## MCP Server is Just a Local Script
An MCP server does NOT need to be hosted. Claude Desktop and Claude Code spawn it as a local subprocess via stdio. The config just points to a binary:
```json
{ "command": "/absolute/path/to/wiki-mcp" }
```
Always use absolute paths — Claude spawns the process in a clean environment where PATH may not include your shell's bin dirs.

## Registering with Claude Code vs Claude Desktop
- Claude Desktop: `~/Library/Application Support/Claude/claude_desktop_config.json`
- Claude Code: `claude mcp add <name> <absolute-path> --scope user` (writes to `~/.claude.json`)
- These are separate configs — you need to register in both.

## tsup Bundling Gotchas
- All source files get bundled into one `dist/index.js`
- `import.meta.dirname` in the bundle = the `dist/` directory, NOT the original source file's directory
- Path calculations using `import.meta.dirname` must account for this:
  - From `dist/index.js`, `../migrations` → `apps/cli/migrations/` ✓
  - `../../migrations` from `dist/` goes two levels up to `apps/` — wrong
- Mark all runtime deps as `--external` in tsup or they won't resolve at runtime

## pnpm Global Linking
```bash
# Must be run from the package directory
cd apps/cli && pnpm link --global
cd apps/mcp && pnpm link --global
```
- Requires `PNPM_HOME` in PATH (run `pnpm setup` first)
- Creates a shell wrapper script, not a symlink
- `import.meta.dirname` resolves to the actual file path, not the wrapper location

## Workspace Package Resolution in Next.js
- `packages/db` exports TypeScript source directly via `"exports": { ".": "./src/index.ts" }`
- Next.js webpack cannot resolve `.js` imports pointing to `.ts` files
- Fix: use `transpilePackages: ["@ai-wiki/db"]` + webpack `extensionAlias`:
```ts
webpack: (config) => {
  config.resolve.extensionAlias = { '.js': ['.ts', '.tsx', '.js'] };
  return config;
}
```

## drizzle-kit Config Gotcha
- `drizzle.config.ts` is loaded by drizzle-kit's own bundler
- Cannot import from workspace TypeScript packages (no compiled output available)
- Inline any config reading logic directly in `drizzle.config.ts` using Node built-ins only
- `drizzle-kit generate` does NOT need a DB connection — only `migrate` does
- Use a try/catch around `getDatabaseUrl()` so generate works without config

## Programmatic Migrations in a CLI
Don't shell out to `pnpm db:migrate` from a setup command — it breaks when:
1. The working directory is wrong
2. The package is installed globally (no monorepo context)

Instead, use drizzle-orm's migrator directly:
```ts
import { migrate } from "drizzle-orm/postgres-js/migrator";
await migrate(db, { migrationsFolder: join(import.meta.dirname, "../migrations") });
```
Output migration files to `apps/cli/migrations/` so they're co-located with the binary.

## Postgres Connection Pool Hangs Process
The `postgres` driver keeps a connection pool open. CLI commands never exit without:
```ts
program.parseAsync().then(() => process.exit(0));
```
Exception: long-running commands like `serve` should not force-exit.

## Config File Pattern for Global CLIs
Store user config at `~/.config/<app>/config.json`, not `.env` in the project dir.
- Works regardless of where the CLI is run from
- Shared between CLI and MCP server automatically
- Still support `DATABASE_URL` env var as override (useful for CI)
21:Tf7c,# dbt Project Structure: Analytics vs AI/ML

## Two Projects, Two Philosophies

### dbt_analytics — Traditional Layered Architecture
Follows the standard dbt staging → intermediate → mart pattern:

```
dbt_analytics/models/
├── staging/           # 1:1 with raw sources, light cleaning only
│   ├── aurora/        # stg_aurora__<table>
│   ├── bi_user/       # stg_bi_user__<table>
│   ├── eventbus/      # stg_eventbus__<table>
│   └── ...
├── intermediate/      # Business logic, domain-grouped
│   ├── profile_engagement/
│   ├── profile_attributes/
│   ├── chat_message/
│   ├── product/
│   └── ...
├── mart/              # Final consumer-facing tables
│   ├── dim_<entity>   # Descriptive attributes
│   └── fact_<event>   # Events, transactions, metrics
└── reports/           # BI-ready aggregations (Looker, etc.)
```

**Strict naming conventions:**
- Staging: `stg_<source>__<table>` (double underscore after source)
- Intermediate: `int__<descriptive_name>` (double underscore prefix, NO source name)
- Mart: `dim_*` for dimensions, `fact_*` for events/transactions (never `mart_*`)
- Reports: domain-specific, often mirrors existing BI tool expectations

### dbt_aiml — Use-Case-Based Architecture
Rather than layers, organized around ML use cases:

```
dbt_aiml/models/
├── studfinder/        # Studfinder recommendation features
├── cascade/           # Cascade click/impression features
├── boostud/           # BoostuD features
├── profile_insights/  # Profile-level analytics for ML
└── ...
```

Each use case deploys to its own Snowflake schema (e.g., `AI_ML.STUDFINDER`). The use-case folder is the primary organizing concept — not whether a model is staging vs. intermediate.

---

## Incremental Patterns

The `delete+insert` strategy became the standard for all incremental models in `dbt_analytics`:

```sql
{{
    config(
        materialized='incremental',
        incremental_strategy='delete+insert',
        unique_key=['ds'],           -- or ['ds', 'hr'] for hourly
        on_schema_change='append_new_columns'
    )
}}
```

**Why `delete+insert` over `merge`:**
- Snowflake `MERGE` locks tables and is slow at scale
- `delete+insert` is faster and aligns with the daily partition model
- `on_schema_change='append_new_columns'` allows schema evolution without full rebuilds

**Hourly models** use `['ds', 'hr']` as the composite unique key and inject both `ds()` and `hr()` Jinja macros for partition filtering.

---

## Source Management

Sources are tracked in `sources_to_dags.yml` files within each project. This file is the contract between dbt models and the upstream Airflow DAGs that populate them:

```yaml
# Example entry in sources_to_dags.yml
- source: bi_user.client_event_daily
  dag_id: client_event_groups
  task_id: tasks_L.BI_USER.CLIENT_EVENT_DAILY
  table: BI_USER.REPORTING.CLIENT_EVENT_DAILY
  monitor_type: daily
```

The pipeline codegen reads this file to automatically generate `ExternalTaskSensor` tasks in the DAG — engineers never manually wire sensors.

---

## Validation via `gdbt validate`

Before any commit, engineers run `gdbt validate` from the dbt project directory. This:

1. Runs SQL linting (`sqlfluff`) with auto-fix
2. Validates task group YAML configs
3. Regenerates all `CODEGEN:` sections in task group YAMLs and DAG configs
4. Regenerates all Airflow DAG Python files from YAML configs
5. Produces a `pipeline_stats.yml` summary

CI runs `gdbt validate --ci` which fails if any files were modified — ensuring local validation was run before pushing.

---

## The `nv/` Folder Convention

Models that haven't yet been fully validated in production are placed in `models/nv/` (non-validated). This allows new models to be deployed without blocking the merge of surrounding migrations. Once validated, they're promoted out of `nv/`.
22:T1138,# Skill System Architecture: File-Based Prompts with a Unified Runner

## Skills Are Files

Each skill is a `SKILL.md` file stored in `.claude/skills/<name>/SKILL.md`. The file has YAML frontmatter followed by the full prompt:

```yaml
---
name: triage
description: Analyze a GitHub issue and propose an implementation approach
allowed-tools: Read, Glob, Grep, Bash(gh:*), Bash(go:list)
---

You are a senior engineer on the Cronos project...
[full prompt follows]
```

The skill file IS the prompt — loaded verbatim at runtime. This makes skills:
- Human-readable (anyone can review what Claude will do)
- Version-controlled (skill evolution is tracked in git history)
- Editable without code changes (no redeployment needed)

## The Unified Runner: `run-skill.sh`

A single script handles local and CI invocation identically:

```bash
.claude/scripts/run-skill.sh /triage 156
.claude/scripts/run-skill.sh /implement 156
.claude/scripts/run-skill.sh /audit 156
```

### What the Runner Does

1. Parses skill name and issue/PR number from args
2. Loads the SKILL.md file from `.claude/skills/<name>/`
3. Determines tool allowlist (READ_TOOLS vs WRITE_TOOLS per skill)
4. Sets `--max-turns` per skill (ask: 10, triage: 20, audit: 30, implement: 40)
5. For **read-only skills**: runs claude with `--print` flag, captures stdout, posts it as a GitHub comment via `gh issue comment`
6. For **write skills**: runs claude normally (Claude has `gh issue comment` access directly)

### The --print Architecture (Critical Bug Fix)

Early versions gave all skills access to `gh issue comment`. This caused a critical bug: triage/audit would accidentally post raw file contents as issue comments (e.g., the full `model_app.go` file appearing as a comment).

Root cause: when Claude had `gh issue comment` access during read operations, it would call it at unexpected times — sometimes with tool output rather than final analysis.

Fix: read-only skills run with `--print` (which outputs final response to stdout and exits). The shell script captures stdout and posts it as the comment. Claude never calls `gh issue comment` directly in read-only mode.

Write skills (`/implement`) still have `gh issue comment` access because they need to post the implementation log mid-run (before the PR exists).

## Tool Allowlists Per Skill

| Skill | Tools |
|-------|-------|
| `/ask` | Read, Glob, Grep, Bash(go list), Bash(gh issue/pr view) |

## Two Distinct AI Roles

### 1. Claude as the Coaching Engine (product feature)
Claude reads structured telemetry data and writes driving coaching briefs. This is an end-user facing feature.

### 2. Claude Code as the Dev Tool (developer tool)
The entire app was built by a single developer using Claude Code, with skills defined in `.claude/skills/`. The coaching skill, the progressive guide system, the metric decomposition — all of it was designed and implemented through AI-assisted development.

---

## How the Coaching Pipeline Works

### Data Flow
```
telemetry (.db SQLite) 
  → ingest_capture.py (Python) 
  → session_analysis.py (per-lap corner scores) 
  → progressive_guide.py (aggregated reference values per corner) 
  → gather-coaching-context (collects everything into one JSON blob) 
  → Claude (writes coaching brief) 
  → saved to DB + shown in web UI
```

### What Claude Receives
The coaching context JSON includes (~12K chars after slimming from 70K):
- Session metadata (car, track, date, num_drivers, session types)
- Per-lap corner analysis: trail_brake_score, line_quality_score, min_speed_kph, gear, coasting_time, speed_ratio
- Per-component breakdowns: brake_taper, connection, rotation_readiness / lateral_g_shape, steering_smoothness, steering_corrections
- Progressive guide reference values (what good looks like for this driver at this track)
- Prior coaching at same track (most recent session)
- Prior coaching cross-track (if different track, for transfer tests)
- Fundamentals (car-level scores, focus areas, history)
- Technique wiki (relevant markdown files)

### Model Choice
- Production: `claude-sonnet-4-6` (v2 "open coach" style)
- Tested: `claude-haiku-4-5` for cost reduction — fast but coaching quality dropped noticeably on complex sessions
- v1 was a "prescriptive" style (more rigid, step-by-step instructions); v2 is an "open coach" (more contextual, adapts to what the driver actually needs)

---

## Key Learnings

### 1. Aggregates are useless for coaching — decompose everything
The first version used `trail_braked_pct` (binary: did the driver coast for less than 0.15s?). The driver repeatedly "failed" trail braking without understanding why. The fix: decompose into 3 sub-components (brake_taper, connection, rotation_readiness), score each 0-100, surface the weakest one. The driver immediately understood: "my taper is 74 but connection is 26 — I'm lifting the brake before touching the throttle."

**Pattern to reuse:** Any metric where the coaching says "improve X" but the driver doesn't know how → decompose into sub-skills, score independently, color-code.

### 2. Token reduction requires curation, not summarization
Initial coaching context was 70K+ chars (raw telemetry dumps). Slimmed to ~12K by:
- Only sending aggregated corner metrics, not raw telemetry arrays
- Limiting prior coaching to the most recent session (not the full history)
- Pre-computing all derived metrics in Python, not leaving calculation to the LLM

**Don't ask Claude to compute from raw data.** Pre-aggregate in Python. Claude's job is synthesis and communication, not math.

### 3. Prompt versioning matters when you change philosophy
Added a `coaching_version` field to distinguish v1 (prescriptive) vs v2 (open coach) prompts. When reviewing progress across sessions, the prior coaching style affects how you interpret what advice was given. Tag your prompts.

### 4. The coaching *chain* is as important as per-session analysis
Single-session coaching is shallow. The most valuable coaching pattern is the **transfer test**: when the driver moves to a new track, compare their fundamentals (coasting %, brake smoothness, steering corrections) against the last session at the previous track. Did the skill transfer or did it reset?

Cross-track coaching also enables catch-up: "Last week at Summit Point you were working on brake-throttle connection. Your first session at Oschersleben shows connection scores of 44 — better than your Summit baseline of 26. That transferred."

### 5. Horizontal coaching before vertical — AI needs explicit instructions for this
Without being explicitly told, Claude will drill into the worst corner in detail. But for beginners, this is wrong: if 3 other corners have worse consistency issues, fixing corner 8's overlap first is wasted effort. The coaching skill prompt explicitly instructs "scan all corners first, get everything to adequate, then drill." This is not intuitive for LLMs without the instruction.

### 6. Mental bandwidth model: why skill gates exist
The prompt explains to Claude *why* horizontal-first coaching works: a beginner's brain is fully occupied with vision, racing line, and braking fundamentals. There is literally no spare mental bandwidth to also process trail braking technique. The coaching should tell the driver: "Make [X] automatic before adding [Y] — not because you're not allowed to, but because your brain can't process both yet." This framing gets better driver buy-in than "you're not at that skill level yet."

### 7. Live overlay coaching requires different feedback than post-session coaching
The sim capture tool shows per-corner feedback in real-time (trail brake score, speed delta vs guide, weakest component). This feedback must be:
- **Immediate** (shown within 2-3 seconds of corner exit)
- **Single metric** (the driver can't read a paragraph mid-session)
- **Actionable in the next lap** (not a historical trend)

Post-session Claude coaching is the opposite: narrative, trend-based, multi-lap patterns.

### 8. Debug logging enables AI-to-AI comparison
The `live_log` table captures exactly what the driver saw during the session (banners, corner scores, next-corner previews). This means you can compare live feedback (what Claude's coaching said to focus on) against post-session offline analysis (what actually happened). Discovered one case where live scores were systematically lower than offline scores due to smoothing kernel difference — only visible because both sides were logged.

- **Static track guides:** Required pre-built reference data per car/track combo. Never scaled. Progressive guide replaced them entirely.

- **Letting Python scripts mutate the `uploads` table:** Node owns that table now. Mixing ownership caused status-update races.
- **Token-heavy contexts:** 70K chars caused slow responses and increased cost. Aggressive pre-aggregation in Python is required.
