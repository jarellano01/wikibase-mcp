18:T81f,# Auto-Registering GraphQL Resolvers via File Naming Conventions

Instead of manually importing and registering every resolver, use a codegen script that scans the filesystem by naming convention and generates a single NestJS resolver module.

## Convention

- `m_*.ts` — mutation resolver
- `q_*.ts` — query resolver
- `report.*.ts` — paginated report query
- Files live in `src/resolvers/crudModels/{entity}/`

## Each resolver file exports a fixed interface

```typescript
@ArgsType()
export class QueryArgs {
  @Field(() => String)
  someField: string;
}

export const OutputType = SomeObjectType;

export const fn = async ({
  context,
  args,
}: {
  context: IContext;
  args: QueryArgs;
}): Promise<typeof OutputType> => {
  await canAccess(context, [perms.resource.action]);
  // ...
};
```

## generateResolvers script

Uses `glob` to find all `[mq]_*.ts` files, extracts resolver name from filename (strips `m_`/`q_` prefix), infers GQL type from prefix, then renders a Handlebars template:

```typescript
const getCrudModelResolvers = async (rootDir: string) => {
  const tsFiles = await glob(`${rootDir}/**/[mq]_*.ts`);
  return tsFiles.map((filePath) => {
    const file = path.parse(filePath);
    const gqlType = file.name[0] === 'q' ? 'Query' : 'Mutation';
    return {
      filePath: getFullPathWithoutExtension(filePath),
      resolverName: file.name.slice(2),
      gqlType,
    };
  });
};
```

## Generated module (Handlebars template)

```handlebars
@Resolver()
class CrudModelsResolver {
  constructor(readonly dbService: DbService) {}

{{#each resolvers}}
  @{{gqlType}}(() => {{resolverName}}.OutputType)
  async {{resolverName}}(
    @Args() args: {{resolverName}}.QueryArgs,
    @Context() context: IContext,
  ) {
    return await {{resolverName}}.fn({ context, args });
  }
{{/each}}
}
```

## Benefits

- Adding a new resolver = create one file, run `pnpm generateResolvers`
- No manual imports or module wiring
- Enforces consistent resolver interface across the codebase
- Easy to see all mutations/queries at a glance from the filesystem
19:T6ac,# Inline Auto-Save Pattern for Form Fields

For forms where each field change should persist immediately (no submit button), this pattern fires a mutation directly in the field's `onChange` handler.

## Core implementation

```typescript
const [isSaving, setIsSaving] = useState(false)
const { runMutation: updateMutation } = useUpdateEntity()

const saveField = useCallback(async (params: Partial<EntityFields>) => {
  if (!id || isSaving) return
  setIsSaving(true)
  try {
    const result = await updateMutation({ variables: { id, ...params } })
    if (result?.success) {
      // Invalidate Apollo cache for related queries
      clearRootQueries(client, ["report_EntityListPage"])
      toast.success("Saved")
      refetchData?.()
    }
  } catch (err: any) {
    toast.error(err?.message || "Failed to save")
  } finally {
    setIsSaving(false)
  }
}, [id, isSaving, updateMutation, client, refetchData])
```

## Field usage

```tsx
<InlineDatePicker
  value={data?.someDate}
  onChange={async (date) => {
    await saveField({ someDate: date })
    // Update local form state without marking it dirty
    form.setValue("someDate", date, { shouldDirty: false })
  }}
  disabled={isLocked || isSaving}
/>
```

## Key details

- **`isSaving` guard**: prevents concurrent saves from the same component
- **`shouldDirty: false`**: keeps RHF from thinking the form has unsaved changes after a successful save
- **Disable inputs while saving**: `disabled={isLocked || isSaving}` prevents race conditions
- **Cache invalidation**: `clearRootQueries` refreshes list pages that show the same data
- **`isLocked` pattern**: separate business logic flag (e.g. invoice is finalized) from save-in-progress flag
1a:T75d,# UUID-Based Permission System with Composite Role Groups

## Permission constants

Permissions are UUID strings organized by resource, defined in `shared-types`:

```typescript
export const perms = {
  invoices: {
    create: "0cde0c13-e5af-40eb-b7df-957620a65043",
    read:   "a33c8e35-2c79-45c8-9f02-b5aace85db22",
    update: "79f4bc16-b8d9-4293-9ffe-9835f9c37860",
    delete: "b1f8c0bb-df00-43db-9e96-0952b9709f63",
    listPageRead: "fc424c5f-60a7-4005-aab7-f95ebd00d17d",
  },
  comments: {
    create: "53fd1293-...",
    // ...
  },
}
```

Using UUIDs (vs. string slugs) means renaming a resource doesn't invalidate existing DB rows.

## Role composition

Base groups bundle related perms, composite groups bundle base groups:

```typescript
const invoicesBaseGroup = {
  perms: [perms.invoices.create, perms.invoices.read, ...],
}

// Composite roles include relevant base groups
const partners = [...invoicesBaseGroup, ...projectsBaseGroup, ...]
const superuser = [...partners, ...adminBaseGroup, ...]
```

## canAccess in resolvers

Every resolver calls `canAccess` as its first line:

```typescript
export const fn = async ({ context, args }) => {
  await canAccess(context, [perms.invoices.update])
  // proceeds only if user has the permission
}
```

`canAccess` accepts an array — passing multiple perms means "any one of these" (OR logic).

## Impersonation support

The system distinguishes `context.user` (authenticated user) from `context.actAsUser` (impersonated user). `canAccess` first checks if the auth user is allowed to impersonate, then checks the act-as user's permissions.

## Benefits

- UUIDs are stable — safe to rename resources without DB migration
- Consistent `listPageRead` permission separates list access from single-record read
- Composite groups make role assignment simple
- Permission cache (10-min TTL) avoids N+1 DB hits per request
1b:Ta31,# How ai-wiki Works

## What It Is
A personal knowledge base designed for AI sessions. Store thoughts, ideas, and articles once — retrieve them in any Claude session on any machine.

## Architecture
```
apps/cli     → wiki CLI (add, search, get, list, delete, setup, serve)
apps/mcp     → local MCP stdio server (Claude uses this automatically)
apps/web     → Next.js read-only dashboard (wiki serve)
packages/db  → Drizzle schema + queries shared by all apps
```
All apps share the same Postgres database.

## Storage
- **Database**: PostgreSQL (Neon, Supabase, Railway, or local)
- **Schema**: `ai_wiki` (isolated from public schema)
- **Table**: `ai_wiki.entries` with fields: id, title, content, summary, tags, type, createdAt, updatedAt
- **Config**: `~/.config/ai-wiki/config.json` stores the DATABASE_URL

## Entry Types
- `note` — general notes
- `idea` — ideas to explore
- `article` — long-form content
- `thought` — quick thoughts

## Token Efficiency
Each entry has an optional `summary` field. MCP tools like `wiki_list` and `wiki_search` return summaries instead of full content — keeping token usage low. Use `wiki_get <id>` to retrieve full content when needed.

## CLI Commands
```bash
wiki setup             # first-time setup: DB config, migrations, MCP registration
wiki add               # interactive prompt to add a new entry
wiki list              # list recent entries (id + title + summary)
wiki search <query>    # keyword search across title, content, summary, tags
wiki get <id>          # retrieve full entry by UUID
wiki delete <id>       # delete an entry (with confirmation)
wiki serve             # start Next.js web dashboard on localhost:3000
```

## MCP Tools (available to Claude automatically)
- `wiki_add` — add an entry
- `wiki_list` — list recent entries (summaries only)
- `wiki_search` — search by keyword
- `wiki_get` — get full entry by ID
- `wiki_delete` — delete an entry

## Setup on a New Machine
```bash
git clone <repo>
pnpm install
pnpm db:generate        # no DB needed
pnpm build
cd apps/cli && pnpm link --global
cd ../mcp && pnpm link --global
source ~/.zshrc
wiki setup              # enter DB URL, runs migrations, registers MCP
claude mcp add ai-wiki /absolute/path/to/wiki-mcp --scope user
```
Restart Claude Desktop if using it. Claude Code picks up the MCP immediately.

## Database Migrations
Always use Drizzle Kit — never manually modify the database.
```bash
# After editing packages/db/src/schema.ts:
pnpm db:generate   # generate migration file
pnpm db:migrate    # apply to DB (or run wiki setup again)
```
1c:T6de,# Reusable CommentBox with conversationKey Namespacing

A single `<CommentBox>` component handles threaded comments anywhere in the app. The thread identity is just a string key — no schema changes needed to add comments to a new entity.

## Usage

```tsx
import { CommentBox } from "@/@containers/dataComponents/CommentBox"

// Convention: "{entity}-{field}-{identifier}"
<CommentBox
  conversationKey="projects-billingNotes-12345"
  title="Billing Notes"
  legacyNote={project.billingNotes
    ? { content: project.billingNotes, label: "Legacy Note" }
    : undefined
  }
/>
```

## Props

```typescript
{
  conversationKey: string          // Thread identity — scopes the comment thread
  title?: string                   // Header label
  subtitle?: string                // Secondary header text
  maxHeight?: string               // CSS max-height for scroll
  legacyNote?: {                   // Read-only pinned note at top (for migrating old free-text fields)
    content: string
    label: string
  }
  showNoLegacyNoteIndicator?: boolean
}
```

## Key features

- **Zero schema work**: adding comments to a new entity = drop in the component with a new key
- **Legacy migration**: `legacyNote` prop shows a read-only pinned note at the top — great for migrating old plain-text fields to the comment system without losing history
- **Keyboard shortcuts**: Cmd+Enter to save, Escape to cancel
- **Owner-only controls**: edit/delete buttons only visible to comment author
- **Markdown rendering** with whitespace preservation
- **"(edited)" indicator** when a comment has been modified

`"{entity}-{field}-{identifier}"` e.g.:
- `"projects-billingNotes-42"`
- `"invoices-internalNotes-99"`
- `"contacts-notes-7"`
