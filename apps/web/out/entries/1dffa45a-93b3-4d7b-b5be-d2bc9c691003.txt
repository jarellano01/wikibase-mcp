16:T75d,# UUID-Based Permission System with Composite Role Groups

## Permission constants

Permissions are UUID strings organized by resource, defined in `shared-types`:

```typescript
export const perms = {
  invoices: {
    create: "0cde0c13-e5af-40eb-b7df-957620a65043",
    read:   "a33c8e35-2c79-45c8-9f02-b5aace85db22",
    update: "79f4bc16-b8d9-4293-9ffe-9835f9c37860",
    delete: "b1f8c0bb-df00-43db-9e96-0952b9709f63",
    listPageRead: "fc424c5f-60a7-4005-aab7-f95ebd00d17d",
  },
  comments: {
    create: "53fd1293-...",
    // ...
  },
}
```

Using UUIDs (vs. string slugs) means renaming a resource doesn't invalidate existing DB rows.

## Role composition

Base groups bundle related perms, composite groups bundle base groups:

```typescript
const invoicesBaseGroup = {
  perms: [perms.invoices.create, perms.invoices.read, ...],
}

// Composite roles include relevant base groups
const partners = [...invoicesBaseGroup, ...projectsBaseGroup, ...]
const superuser = [...partners, ...adminBaseGroup, ...]
```

## canAccess in resolvers

Every resolver calls `canAccess` as its first line:

```typescript
export const fn = async ({ context, args }) => {
  await canAccess(context, [perms.invoices.update])
  // proceeds only if user has the permission
}
```

`canAccess` accepts an array — passing multiple perms means "any one of these" (OR logic).

## Impersonation support

The system distinguishes `context.user` (authenticated user) from `context.actAsUser` (impersonated user). `canAccess` first checks if the auth user is allowed to impersonate, then checks the act-as user's permissions.

## Benefits

- UUIDs are stable — safe to rename resources without DB migration
- Consistent `listPageRead` permission separates list access from single-record read
- Composite groups make role assignment simple
- Permission cache (10-min TTL) avoids N+1 DB hits per request
17:T7c0,# URL-Synced Filter & Pagination State with nuqs

Each list page has a co-located `params.ts` that owns all filter/sort/pagination state in the URL via `nuqs`.

## File structure

```
src/app/(dashboard)/some-list/
├── params.ts       ← owns all URL state
├── filterBar.tsx   ← reads/writes params
└── page.tsx        ← composes FilterBar + table
```

## params.ts pattern

```typescript
"use client"

import { parseAsArrayOf, parseAsInteger, parseAsJson, parseAsString, useQueryStates } from "nuqs"
import { useMemo } from "react"
import { z } from "zod"

const sortSchema = z.array(z.object({ id: z.string(), desc: z.boolean() }))

export const useParams = () => {
  const [params, setParams] = useQueryStates(
    {
      page: parseAsInteger.withDefault(1),
      searchTerm: parseAsString,
      status: parseAsArrayOf(parseAsString).withDefault(["active"]),
      sort: parseAsJson(sortSchema.parse).withDefault([{ id: "createdAt", desc: true }]),
      limit: parseAsInteger.withDefault(50),
    },
    {
      urlKeys: { limit: "perPage" }, // rename URL key without changing JS name
    },
  )

  const updatedParams = useMemo(() => ({
    ...params,
    searchTerm: params.searchTerm ?? undefined, // null → undefined for optional GQL args
    pageIndex: params.page - 1,                 // convert 1-based page to 0-based index
  }), [params])

  return [updatedParams, setParams] as const
}
```

## Key ideas

- **Shareable URLs**: filters survive page refresh and can be copied/bookmarked
- **Type-safe parsers**: `parseAsInteger`, `parseAsString`, `parseAsArrayOf`, `parseAsJson` with Zod validation
- **`.withDefault()`**: guarantees non-null value so downstream code doesn't need null checks
- **`urlKeys`**: rename the URL param without changing the JS variable name
- **null → undefined transform**: nuqs uses `null` for absent values; GraphQL args expect `undefined`
- Returns `[params, setParams]` tuple following React hook convention
18:Te6a,# Building a CLI + MCP Server in a TypeScript Monorepo

## Architecture
- Turborepo monorepo with pnpm workspaces
- `packages/db` — shared Drizzle ORM schema, queries, config
- `apps/cli` — Commander.js CLI (`wiki` binary)
- `apps/mcp` — local MCP stdio server (`wiki-mcp` binary)
- `apps/web` — Next.js read-only dashboard

## MCP Server is Just a Local Script
An MCP server does NOT need to be hosted. Claude Desktop and Claude Code spawn it as a local subprocess via stdio. The config just points to a binary:
```json
{ "command": "/absolute/path/to/wiki-mcp" }
```
Always use absolute paths — Claude spawns the process in a clean environment where PATH may not include your shell's bin dirs.

## Registering with Claude Code vs Claude Desktop
- Claude Desktop: `~/Library/Application Support/Claude/claude_desktop_config.json`
- Claude Code: `claude mcp add <name> <absolute-path> --scope user` (writes to `~/.claude.json`)
- These are separate configs — you need to register in both.

## tsup Bundling Gotchas
- All source files get bundled into one `dist/index.js`
- `import.meta.dirname` in the bundle = the `dist/` directory, NOT the original source file's directory
- Path calculations using `import.meta.dirname` must account for this:
  - From `dist/index.js`, `../migrations` → `apps/cli/migrations/` ✓
  - `../../migrations` from `dist/` goes two levels up to `apps/` — wrong
- Mark all runtime deps as `--external` in tsup or they won't resolve at runtime

## pnpm Global Linking
```bash
# Must be run from the package directory
cd apps/cli && pnpm link --global
cd apps/mcp && pnpm link --global
```
- Requires `PNPM_HOME` in PATH (run `pnpm setup` first)
- Creates a shell wrapper script, not a symlink
- `import.meta.dirname` resolves to the actual file path, not the wrapper location

## Workspace Package Resolution in Next.js
- `packages/db` exports TypeScript source directly via `"exports": { ".": "./src/index.ts" }`
- Next.js webpack cannot resolve `.js` imports pointing to `.ts` files
- Fix: use `transpilePackages: ["@ai-wiki/db"]` + webpack `extensionAlias`:
```ts
webpack: (config) => {
  config.resolve.extensionAlias = { '.js': ['.ts', '.tsx', '.js'] };
  return config;
}
```

## drizzle-kit Config Gotcha
- `drizzle.config.ts` is loaded by drizzle-kit's own bundler
- Cannot import from workspace TypeScript packages (no compiled output available)
- Inline any config reading logic directly in `drizzle.config.ts` using Node built-ins only
- `drizzle-kit generate` does NOT need a DB connection — only `migrate` does
- Use a try/catch around `getDatabaseUrl()` so generate works without config

## Programmatic Migrations in a CLI
Don't shell out to `pnpm db:migrate` from a setup command — it breaks when:
1. The working directory is wrong
2. The package is installed globally (no monorepo context)

Instead, use drizzle-orm's migrator directly:
```ts
import { migrate } from "drizzle-orm/postgres-js/migrator";
await migrate(db, { migrationsFolder: join(import.meta.dirname, "../migrations") });
```
Output migration files to `apps/cli/migrations/` so they're co-located with the binary.

## Postgres Connection Pool Hangs Process
The `postgres` driver keeps a connection pool open. CLI commands never exit without:
```ts
program.parseAsync().then(() => process.exit(0));
```
Exception: long-running commands like `serve` should not force-exit.

## Config File Pattern for Global CLIs
Store user config at `~/.config/<app>/config.json`, not `.env` in the project dir.
- Works regardless of where the CLI is run from
- Shared between CLI and MCP server automatically
- Still support `DATABASE_URL` env var as override (useful for CI)
19:T1696,# Pipeline Framework: YAML-to-Airflow Codegen

## Core Idea

Engineers should never need to write Airflow Python to add a new dbt pipeline. They configure what runs in YAML files inside the dbt project, run `gdbt validate`, and the framework generates correct, production-grade Airflow DAG Python files.

This decoupling means:
- Data engineers work entirely in the dbt repo
- DAG Python files are treated as build artifacts, not source code
- Sensor wiring, task groups, and scheduling are derived from the dbt model graph

---

## Config Hierarchy

```
dbt_analytics/pipelines/
├── config.yaml                    # Project-level settings
├── dags/                          # One YAML per Airflow DAG
│   ├── profile_engagement.yml
│   ├── product_analytics.yml
│   └── ...
└── task_groups/                   # One YAML per logical task group
    ├── intermediate/
    │   ├── profile_engagement.yml
    │   ├── product.yml
    │   └── profile_attributes/
    │       ├── profile_device.yml
    │       └── ...
    ├── mart/
    └── staging/
```

### DAG Config (`dags/<name>.yml`)

```yaml
name: profile_engagement
description: Daily pipeline for profile engagement models.

output_dag: dags/dbt/analytics/profile_engagement.py
schedule: daily 18:00
start_date: 2026-03-08
catchup: true
depends_on_past: true

config:
  dbt_project: analytics
  warehouse:
    local: DE_ETL_WH
    prod: DE_ETL_WH

task_groups:
- file: intermediate/profile_engagement.yml
- file: mart/profile_engagement.yml

# CODEGEN sections below — do not edit manually
```

### Task Group Config (`task_groups/<layer>/<name>.yml`)

```yaml
name: profile_engagement
dbt:
  selector: path:models/intermediate/profile_engagement

# CODEGEN: Auto-populated by gdbt validate
included_models:
- key: int__profile_taps_daily
  type: incremental
  target: BI_USER.REPORTING.PROFILE_TAPS_DAILY

upstream_task_groups:
- intermediate/product

sources:
- bi_user.client_event_daily
- eventbus.login_event
```

---

## Codegen Process

When `gdbt validate` runs, the Python codegen layer (`generate_dag.py`, `generate_pipeline.py`) does:

1. **Parse task group YAMLs** — load `dbt` selectors, extract included models by running `dbt ls`
2. **Resolve upstream dependencies** — for each source in a task group, look up `sources_to_dags.yml` to find the upstream DAG + task ID
3. **Compute execution deltas** — given the downstream DAG schedule and upstream DAG schedule, calculate the `timedelta` for each `ExternalTaskSensor`
4. **Populate CODEGEN sections** — write auto-discovered sensors, upstream DAGs, downstream DAGs, and stats back into the YAML files with `# CODEGEN:` comment markers
5. **Generate Airflow Python DAG** — render `dag_template.jinja2` with all config, producing a fully valid Airflow Python file
6. **Validate references** — check that all referenced task group files exist, all sensor external_dag_ids exist

The `# CODEGEN:` comment is the contract: anything below it in a YAML or Python file is machine-written and will be overwritten on the next `gdbt validate --fix`.

---

## Generated DAG Structure

The generated Python DAG always follows the same structure:

```python
# CODEGEN: This file is code-generated and should not be modified manually
# Generated from: dbt_analytics/pipelines/dags/profile_engagement.yml
# To regenerate: cd dbt_analytics && gdbt validate

from cosmos import DbtTaskGroup, ProjectConfig, RenderConfig
from plugins.util import pipeline_builder as pb

SNOWFLAKE_WAREHOUSE = pb.utils.get_warehouse_for_env(local_warehouse='DE_ETL_WH', prod_warehouse='DE_ETL_WH')

dag_config = pb.config.DagConfig(dag_id='profile_engagement', ...)

with DAG(**dag_config.to_dag_kwargs()) as dag:
    # For each task group:
    with TaskGroup('intermediate_profile_engagement') as tg:
        with TaskGroup('source_sensors') as _sensors:
            ExternalTaskSensor(task_id='wait_for_...', ...)
        
        DbtTaskGroup(
            group_id='dbt_models',
            project_config=ProjectConfig(...),
            render_config=RenderConfig(select=['path:models/intermediate/profile_engagement'], ...),
            ...
        )
```

**Astronomer Cosmos** (`DbtTaskGroup`) handles the actual dbt model execution within Airflow, providing per-model task visibility in the Airflow UI.

---

## Sensor Framework

The sensor system evolved significantly:

**v1 (manual):** Engineers wrote `ExternalTaskSensor` directly in DAG Python files

**v2 (sources_to_dags.yml):** Sources declared in YAML; sensors auto-generated from the file

**v3 (task group sensors):** Sensors scoped to the task group that needs them, not the DAG level

**v4 (sensor overrides, DATA-4545):** Per-task-group `sensor_config` in YAML allows overriding sensor type, retries, and execution delta without touching codegen:

```yaml
# In a task group YAML, under sensor_config
sensor_config:
  freshness_check_override:
    source: chat_sensitive.chat_platform
    table: CHAT_SENSITIVE.REPORTING.CHAT_PLATFORM
    monitor_type: hourly
  external_task_sensor_override:
    external_dag_id: some_other_dag
    external_task_group_id: specific_task_group
```

**Centralized source sensors (DATA-4583):** A dedicated `dbt_shared_source_sensors` DAG performs freshness checks on all shared upstream tables. Domain DAGs wait on this DAG rather than each independently checking the same sources.

1a:Ta31,# How ai-wiki Works

## What It Is
A personal knowledge base designed for AI sessions. Store thoughts, ideas, and articles once — retrieve them in any Claude session on any machine.

## Architecture
```
apps/cli     → wiki CLI (add, search, get, list, delete, setup, serve)
apps/mcp     → local MCP stdio server (Claude uses this automatically)
apps/web     → Next.js read-only dashboard (wiki serve)
packages/db  → Drizzle schema + queries shared by all apps
```
All apps share the same Postgres database.

## Storage
- **Database**: PostgreSQL (Neon, Supabase, Railway, or local)
- **Schema**: `ai_wiki` (isolated from public schema)
- **Table**: `ai_wiki.entries` with fields: id, title, content, summary, tags, type, createdAt, updatedAt
- **Config**: `~/.config/ai-wiki/config.json` stores the DATABASE_URL

## Entry Types
- `note` — general notes
- `idea` — ideas to explore
- `article` — long-form content
- `thought` — quick thoughts

## Token Efficiency
Each entry has an optional `summary` field. MCP tools like `wiki_list` and `wiki_search` return summaries instead of full content — keeping token usage low. Use `wiki_get <id>` to retrieve full content when needed.

## CLI Commands
