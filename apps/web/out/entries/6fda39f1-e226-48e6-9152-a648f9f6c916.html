Triage Skill: Readiness Signal Pattern

The Pattern

Every /triage output ends with a readiness signal — a traffic-light assessment of whether the issue is ready to implement:

  • 🟢 Green — Clear scope, low risk, localized changes, existing tests cover the area. Implement immediately.
  • 🟡 Yellow — Moderate uncertainty, touches shared logic, edge cases unclear. Review triage output first, possibly run /audit.
  • 🔴 Red — Affects accounting/GL, requires migrations, state machine changes, scope too vague. Run /audit first; consider splitting the issue.

Why It Works

Without a readiness signal, developers tend to start implementing before the scope is clear. The traffic light forces an explicit decision gate: "Are we confident enough to proceed?" This stopped a pattern of implementation work starting on under-scoped or too-large issues.

Triage Output Structure

## Affected Areas
- Models: [list]
- Files: [list]
- State Transitions: [if any]
- Risk Areas: [if any]

## Proposed Approach
1. [Concrete bullet]
2. [Concrete bullet]
...

## Testing Strategy
[What tests to write/update]

## Related Issues
[Links if any]

## Open Questions
[Blockers or important unknowns]

## Readiness: 🟢 Green
[One-line rationale]

Escalation Path

  • Green → go straight to /implement
  • Yellow → discuss in comments, possibly /audit, then /implement
  • Red → always run /audit; if audit finds it's too large, break into smaller issues, then triage/implement each

Max Turns

The triage skill runs with --max-turns 20. This is enough for most issues (scoped analysis of 3-10 files). If triage hits the limit, it's a signal the issue is too large and should be escalated to /audit (which runs at --max-turns 30).

Contrast with Audit

Triage is prescriptive — it identifies the problem and proposes a specific approach. Audit is exploratory — it surfaces trade-offs and options without prescribing. Use triage by default; escalate to audit only when triage returns red or the issue is explicitly architectural.
