8c:T19f1,# HotLap: How AI Is Used and What Was Learned

## Two Distinct AI Roles

### 1. Claude as the Coaching Engine (product feature)
Claude reads structured telemetry data and writes driving coaching briefs. This is an end-user facing feature.

### 2. Claude Code as the Dev Tool (developer tool)
The entire app was built by a single developer using Claude Code, with skills defined in `.claude/skills/`. The coaching skill, the progressive guide system, the metric decomposition — all of it was designed and implemented through AI-assisted development.

---

## How the Coaching Pipeline Works

### Data Flow
```
telemetry (.db SQLite) 
  → ingest_capture.py (Python) 
  → session_analysis.py (per-lap corner scores) 
  → progressive_guide.py (aggregated reference values per corner) 
  → gather-coaching-context (collects everything into one JSON blob) 
  → Claude (writes coaching brief) 
  → saved to DB + shown in web UI
```

### What Claude Receives
The coaching context JSON includes (~12K chars after slimming from 70K):
- Session metadata (car, track, date, num_drivers, session types)
- Per-lap corner analysis: trail_brake_score, line_quality_score, min_speed_kph, gear, coasting_time, speed_ratio
- Per-component breakdowns: brake_taper, connection, rotation_readiness / lateral_g_shape, steering_smoothness, steering_corrections
- Progressive guide reference values (what good looks like for this driver at this track)
- Prior coaching at same track (most recent session)
- Prior coaching cross-track (if different track, for transfer tests)
- Fundamentals (car-level scores, focus areas, history)
- Technique wiki (relevant markdown files)

### Model Choice
- Production: `claude-sonnet-4-6` (v2 "open coach" style)
- Tested: `claude-haiku-4-5` for cost reduction — fast but coaching quality dropped noticeably on complex sessions
- v1 was a "prescriptive" style (more rigid, step-by-step instructions); v2 is an "open coach" (more contextual, adapts to what the driver actually needs)

---

## Key Learnings

### 1. Aggregates are useless for coaching — decompose everything
The first version used `trail_braked_pct` (binary: did the driver coast for less than 0.15s?). The driver repeatedly "failed" trail braking without understanding why. The fix: decompose into 3 sub-components (brake_taper, connection, rotation_readiness), score each 0-100, surface the weakest one. The driver immediately understood: "my taper is 74 but connection is 26 — I'm lifting the brake before touching the throttle."

**Pattern to reuse:** Any metric where the coaching says "improve X" but the driver doesn't know how → decompose into sub-skills, score independently, color-code.

### 2. Token reduction requires curation, not summarization
Initial coaching context was 70K+ chars (raw telemetry dumps). Slimmed to ~12K by:
- Only sending aggregated corner metrics, not raw telemetry arrays
- Limiting prior coaching to the most recent session (not the full history)
- Pre-computing all derived metrics in Python, not leaving calculation to the LLM

**Don't ask Claude to compute from raw data.** Pre-aggregate in Python. Claude's job is synthesis and communication, not math.

### 3. Prompt versioning matters when you change philosophy
Added a `coaching_version` field to distinguish v1 (prescriptive) vs v2 (open coach) prompts. When reviewing progress across sessions, the prior coaching style affects how you interpret what advice was given. Tag your prompts.

### 4. The coaching *chain* is as important as per-session analysis
Single-session coaching is shallow. The most valuable coaching pattern is the **transfer test**: when the driver moves to a new track, compare their fundamentals (coasting %, brake smoothness, steering corrections) against the last session at the previous track. Did the skill transfer or did it reset?

Cross-track coaching also enables catch-up: "Last week at Summit Point you were working on brake-throttle connection. Your first session at Oschersleben shows connection scores of 44 — better than your Summit baseline of 26. That transferred."

### 5. Horizontal coaching before vertical — AI needs explicit instructions for this
Without being explicitly told, Claude will drill into the worst corner in detail. But for beginners, this is wrong: if 3 other corners have worse consistency issues, fixing corner 8's overlap first is wasted effort. The coaching skill prompt explicitly instructs "scan all corners first, get everything to adequate, then drill." This is not intuitive for LLMs without the instruction.

### 6. Mental bandwidth model: why skill gates exist
The prompt explains to Claude *why* horizontal-first coaching works: a beginner's brain is fully occupied with vision, racing line, and braking fundamentals. There is literally no spare mental bandwidth to also process trail braking technique. The coaching should tell the driver: "Make [X] automatic before adding [Y] — not because you're not allowed to, but because your brain can't process both yet." This framing gets better driver buy-in than "you're not at that skill level yet."

### 7. Live overlay coaching requires different feedback than post-session coaching
The sim capture tool shows per-corner feedback in real-time (trail brake score, speed delta vs guide, weakest component). This feedback must be:
- **Immediate** (shown within 2-3 seconds of corner exit)
- **Single metric** (the driver can't read a paragraph mid-session)
- **Actionable in the next lap** (not a historical trend)

Post-session Claude coaching is the opposite: narrative, trend-based, multi-lap patterns.

### 8. Debug logging enables AI-to-AI comparison
The `live_log` table captures exactly what the driver saw during the session (banners, corner scores, next-corner previews). This means you can compare live feedback (what Claude's coaching said to focus on) against post-session offline analysis (what actually happened). Discovered one case where live scores were systematically lower than offline scores due to smoothing kernel difference — only visible because both sides were logged.

---

## What Didn't Work

- **Static track guides:** Required pre-built reference data per car/track combo. Never scaled. Progressive guide replaced them entirely.
- **`race_events` table:** Awkward date-based grouping. Uploads are a better container. Slated for removal.
- **Letting Python scripts mutate the `uploads` table:** Node owns that table now. Mixing ownership caused status-update races.
- **Token-heavy contexts:** 70K chars caused slow responses and increased cost. Aggressive pre-aggregation in Python is required.
8d:T1372,# HotLap: iRacing Coaching App — Product Overview and Evolution

## What It Is

HotLap is a personal iRacing coaching app built for one driver (the developer, a beginner). It closes the loop between sim racing sessions and structured skill improvement:

1. A **Windows capture tool** (PyInstaller .exe) runs alongside iRacing, captures telemetry to a SQLite `.db` file, and shows a real-time overlay with per-corner feedback.
2. After a session, the capture tool uploads the `.db` to GCS.
3. A **Next.js web app** (Cloud Run) ingests the file via Python ML scripts, runs analysis, and stores structured results in PostgreSQL (Neon).
4. The web app shows a **progressive guide** — a per-corner breakdown of the driver's technique built from their own telemetry across sessions.
5. A **Claude-powered coaching skill** reads all that data and writes a structured coaching brief.

**Stack:** Next.js 16 (App Router, React 19) + Python ML pipeline. Node spawns Python scripts via `child_process.spawn`. DB is Neon PostgreSQL with Drizzle ORM.

## Real Usage Data (as of March 2026)

- **63 sessions** ingested across 7 tracks
- Primary track: **Summit Point Raceway** — 22 sessions, 190 laps, best lap 1:22.267
- Second most-practiced: **Okayama International Circuit** — 22 sessions, 110 laps
- Primary car: Mazda MX-5 Cup
- The progressive guide at Summit Point has **8 corners** with 21 sessions of data — well past the "stable" confidence threshold (5+ sessions)

- Initial scaffold: Next.js + Cloud Run + Auth0
- Python and uv added to Docker image
- Sim-capture build workflow (GitHub Actions → PyInstaller .exe → GCS)
- Basic session ingest, static track guides from Lovely Track Data + TrackTitan

- Corner names from Lovely Track Data shown in overlay

- 29+ bug-fix commits on a single day (the sim capture tool went through v0.2.0–v0.3.2 in one day)

### Week 2 (March 8-16): Architecture Shift — Progressive Guide

The biggest architectural change: **replaced static track guides with a progressive guide** built from the driver's own telemetry.

**Why:** Static guides require pre-built reference data. Progressive guides learn from the driver's actual observations across sessions — no cold-start dependency, and the reference speeds reflect what *this driver in this car* can achieve, not a generic benchmark.

**Key design:** Each corner observation is evaluated independently. A lap where the driver practices T6-T7-T8 and coasts through T1 contributes high-quality data for T6-T7-T8 and nothing for T1. The guide learns from each corner's best observations, not from "best laps."

Confidence gates: learning (<3 sessions), developing (3-5 sessions), stable (5+).

### March 9-10: Metric Decomposition

Trail brake metric evolved from binary (`trail_braked_pct`) → composite 0-100 score with 3 sub-components:
- **brake_taper** (35%): smoothness of brake release through the corner
- **connection** (35%): no dead pedal gap + brief light overlap
- **rotation_readiness** (30%): whether the car is rotated before throttle application

Line quality similarly decomposed into: `lateral_g_shape`, `steering_smoothness`, `steering_corrections`.

This was the most impactful coaching improvement — instead of "your trail braking is weak," the coach could say "your taper is fine at 74 but connection is 26 — you're lifting the brake before getting on throttle."

### March 15-16: New Tracks + Deprecating Fundamentals

Started practicing Motorsport Arena Oschersleben (6 sessions). Cross-track coaching chain established — prior coaching from Summit Point carries into new track sessions as a "transfer test."

Fundamentals system being deprecated in favor of inline skill stage computed from progressive guide data.

## Core Coaching Philosophy (Documented in CLAUDE.md)

Two key principles emerged from real usage and feedback:

1. **Decompose metrics into actionable components.** Binary pass/fail metrics hide which sub-skill is lacking. Every aggregate metric should become a composite score with independently-scored components, color-coded (green ≥70 / yellow ≥40 / red <40), with the weakest component surfaced.

2. **Horizontal first, then drill.** Scan ALL corners before going deep on any one. A driver with inconsistent brake points across 4 corners should not be coached on brake-throttle overlap at corner 8. Get everything to "adequate for your level" first.

Both principles were discovered through real frustration — the driver not understanding coaching feedback and not improving — not from upfront design.

## Session Progression by Track

```sql
-- Best lap, total sessions, total laps per track
SELECT 
  t.name as track,
  MIN(l.lap_time) as best_lap,
  COUNT(DISTINCT s.id) as sessions,
  COUNT(l.id) as total_laps
FROM sim_racing.sessions s
JOIN sim_racing.tracks t ON s.track_id = t.id
JOIN sim_racing.laps l ON l.session_id = s.id
WHERE l.lap_time > 30
GROUP BY t.name
ORDER BY sessions DESC;
```

8f:T108a,# dbt Migration: Cutover Pattern & Zero-Downtime Strategy

## The Two-Phase Approach

Migrating data pipelines without downtime requires running old and new in parallel for a validation window. The pattern used here:

**Phase 1 — Build (dbt models created, not yet in production)**
- New dbt models land in `dbt_analytics/models/` 
- Task group YAML config written
- Tables materialized in `_DEV` database during development
- Old Airflow DAG continues running unchanged
- Human review of model logic, schema alignment, data parity

**Phase 2 — Cutover (both pipelines run, then old retired)**
- Cutover DAG runs both old Airflow SQL and new dbt pipeline
- Column mapping validation confirms output equivalence
- After N successful runs in production, old pipeline tasks removed
- Old Airflow DAG archived or paused

---

## Cutover DAG Structure

Generated by the `/create-cutover-dag` skill, the cutover DAG follows a standard pattern:

```python
# Example: ad_data_model -> ads_analytics cutover
dag_id = 'cutover_ad_data_model_to_ads_analytics'

with DAG(...) as dag:
    # 1. Upstream sensors (same as old DAG)
    wait_for_upstream = ExternalTaskSensor(...)
    
    # 2. Run OLD pipeline (existing Airflow SQL)
    old_pipeline = TaskGroup('old_ad_data_model')
    
    # 3. Run NEW dbt pipeline (cosmos DbtTaskGroup)
    new_pipeline = TaskGroup('new_ads_analytics')
    
    # 4. Validation: compare column counts, row counts
    validate_parity = SnowflakeOperator(
        sql="""
        SELECT 
            'old' as source, COUNT(*) as row_count FROM BI_USER.REPORTING.AD_DATA_MODEL WHERE ds = '{{ ds }}'
            UNION ALL
            SELECT 'new', COUNT(*) FROM BI_USER_DEV.REPORTING.ADS_ANALYTICS WHERE ds = '{{ ds }}'
        """
    )
    
    wait_for_upstream >> [old_pipeline, new_pipeline] >> validate_parity
```

The two pipelines run in parallel after their shared upstream sensor, then both feed into a parity validation step. A discrepancy in the validation step fails the DAG and pages the team.

---

## What Gets Validated

The `/validate-tables-dbt` skill performs:

1. **Row count match** — total rows per `ds` partition must be within tolerance
2. **Column presence** — all columns from the old table exist in new table
3. **Null rates** — key non-null columns don't gain nulls
4. **Aggregate sums** — SUM of metric columns matches (within float tolerance)
5. **Distinct value sets** — for dimension columns, same set of values

Snowflake `query_id`s are posted to the PR so reviewers can re-run the comparison themselves.

## Sensor Migration Strategy

A major coordination challenge: old DAGs use task-level `ExternalTaskSensor` pointing at specific Airflow task IDs. New dbt DAGs use `DbtTaskGroup` task IDs generated by Cosmos, which follow a different naming convention.

**Solution:** When an old DAG has sensors pointing at it, the cutover updates the new DAG's task group ID to match the expected external task ID format, or the downstream DAG's sensor `external_task_group_id` is updated to point at the new dbt task group.

This was handled by the sensor override system (DATA-4545) which allows YAML-level overrides without touching the codegen output.

- The `schema` dbt config maps the intermediate model to the right output schema

Every `/implement` run posts a structured comment on the issue before creating the PR. This "implementation log" is mandatory — it is never skipped even for trivial changes.

## Implementation Log Structure

## Why It Matters

1. **Handoff record** — When a future session picks up the issue, this log provides context that isn't in the code diff
2. **Reviewer context** — PR reviewers can see what option was chosen and why, without re-reading the full triage thread
3. **Searchable history** — The implementation log is on the issue, which stays searchable even after the PR is merged
4. **Surfaces surprises** — "Key Decisions / Findings" is where unexpected bugs or constraints surface before the PR review

## The 10-Step Implement Process

1. Load project context (business docs, CLAUDE.md)
2. Read issue + ALL comments (triage decisions, audit findings, answers to blockers)
3. Verify repo setup: `go build ./...`, `go test . -count=1`, document pre-existing failures
4. Identify blockers (missing AutoMigrate, external service calls, accounting implications)
5. Implement (follow existing patterns exactly; match fixture chains from closest test)
6. Run tests: targeted first, then full suite for regressions
7. **Post implementation log**
8. Commit to feature branch (`feat/issue-N-short` or `fix/issue-N-short`)
9. **ALWAYS create draft PR** (never skip — this was a recurring bug in early versions)
10. Post PR link on the issue

In early versions of the skill, the implement flow would complete after step 8 (commit) and exit before creating the PR. This happened because the skill prompt didn't explicitly emphasize step 9. The fix: add "ALWAYS attempt `gh pr create` — do not skip this step even if you think you're done" with prominent formatting in the skill prompt.
