Audit Skill: Deep Exploratory Analysis for Complex Issues

article · workflow, ai, audit, skills, cronos, analysis · 3/21/2026
The /audit skill does deep multi-dimensional codebase analysis for complex issues. It surfaces bugs the team hadn't noticed, proposes phased implementation plans, and explicitly checks if the issue is too large to implement as-is.

Audit Skill: Deep Exploratory Analysis for Complex Issues

When to Use

Run /audit instead of /triage when:

  • Triage returned 🔴 Red
  • The issue explicitly touches architecture (models, state machines, accounting, multi-tenancy)
  • The scope is unclear and may be too large for a single implementation
  • You want trade-offs and options, not just a prescribed approach

What It Does

Audit performs a structured multi-dimensional analysis:

Seven Audit Dimensions

  1. Data Model Audit — All relevant models, relationships, GORM hooks. What's stored, what's derived.
  2. Business Logic Audit — Full code path trace from handler to DB. Error handling, side effects, edge cases.
  3. State Machine Audit — All valid transitions, gaps, edge cases. Is every state reachable? Every exit valid?
  4. Accounting Audit — Double-entry balance invariants, journal lifecycle. Do debits equal credits in all paths?
  5. Test Coverage Audit — What's tested, what's not. N+1 risks, missing state transitions in tests.
  6. Performance Audit — N+1 queries, unnecessary preloads, missing indexes, pagination gaps.
  7. Security/Integrity Audit — Input validation, auth/authz, multi-tenant scoping.

Fallback to Triage

If the issue turns out to be simple (after fetching context), audit falls back to triage format. No wasted depth on easy problems.

Surfacing Bugs

In practice, audit on complex issues has surfaced bugs the team hadn't noticed:

  • The Bill.TotalAdjustments orphaned field (never updated in production code)
  • Commission truncation bug (integer math on percentage × invoice total)

Output: Phased Implementation Plan

Audit ends with an ordered implementation plan where each task is scoped to be its own GitHub issue:

## Implementation Plan

### Phase 1: Prep (low risk, can merge independently)
- [ ] Task A: [description] — affects [files]
- [ ] Task B: [description] — affects [files]

### Phase 2: Core Changes (medium risk)
- [ ] Task C: [description] — depends on A, B
- [ ] Task D: [description] — accounting event changes

### Phase 3: Cleanup (after Phase 2 proven in production)
- [ ] Task E: [description]

Splitting Issues

If audit determines the issue is too large, it explicitly says so and describes how to break it into child issues. The team creates those child issues manually, then triages/implements each independently.

The Conversational Loop

Audit is designed for iteration:

  1. Run @claude /audit
  2. Read the output. Post follow-up questions or decisions in comments.
  3. Re-run @claude /audit — it reads all prior comments and refines the analysis.
  4. Repeat until the implementation plan feels right.

This conversational loop (on issue #136) demonstrated genuine collaborative analysis: the team asked targeted follow-up questions and got refined answers that materially changed the implementation plan.

Max Turns: 30

Audit runs with --max-turns 30 (vs triage at 20). This is necessary — a thorough audit of a 25+ file codebase was hitting the 15-turn limit (the original value) 60% through the analysis. The increase to 30 resolved this.
