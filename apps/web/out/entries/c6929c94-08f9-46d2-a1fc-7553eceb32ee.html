dbt Patterns: Lessons & Reusable Ideas from the Migration

article · dbt, patterns, lessons, best-practices, ai, codegen, developer-experience · 3/21/2026
Transferable patterns and lessons from building a large-scale Airflow-to-dbt migration with AI assistance: what worked, what didn't, and what any team could borrow.

dbt Patterns: Lessons & Reusable Ideas from the Migration

Pattern 1: CODEGEN Comment Sections as a Contract

Splitting YAML and Python files into "human-written" and "machine-written" sections using # CODEGEN: comment markers turned out to be one of the most durable architectural decisions.

  • Engineers own everything above the marker
  • gdbt validate owns everything below it
  • Merge conflicts in codegen sections are always resolved by accepting deletions and regenerating
  • The /merge-main skill handles this automatically

Why it works: It eliminates an entire class of merge conflicts. Two engineers can both modify the same task group YAML, and as long as neither touches the CODEGEN section, there's no conflict. The codegen regeneration on the next gdbt validate reconciles both changes.

Pattern 2: Encode Review Standards into AI Rules Before Writing Code

The Cursor AI code review rules (.cursor/rules/) were written at the start of the dbt migration, not after. This meant:

  • Every model generated by Cursor was already checked against standards
  • Review comments shrunk from "you need to add X" to "looks good, minor nit"
  • New team members got standards enforcement before they'd read the docs

The lesson: invest in AI review rules as early infrastructure, not a cleanup step. The ROI compounds because every PR in the project benefits.

Pattern 3: Skills for "Journeys", Not Individual Operations

The initial attempt at skills was one skill per operation: /analyze-dag, /generate-models, etc. These were useful but required the engineer to orchestrate them manually.

The breakthrough was the orchestrator skill pattern: /create-dbt-migration composes sub-skills via the Task tool (multi-agent), so engineers get the whole journey in one command. The sub-skills remain individually usable when you need to re-run just one step.

Takeaway: design skills at the level of what the user wants to accomplish, then compose smaller skills inside. The user's mental model is "migrate this DAG", not "run analyze, then generate, then validate".

Pattern 4: Git History as a Quality Signal

Because every commit was AI-assisted (either Cursor auto-applying fixes or Claude Code generating migrations), commit quality became unusually high:

  • Structured ticket references: DATA-XXXX: <verb> <thing>
  • Minimal "fix typo" / "oops" commits (the AI caught these before commit)
  • Codegen regeneration commits cleanly labeled Regenerate codegen files after merge from main
  • Work-in-progress commits clearly labeled WIP:

This made git log --grep searches highly reliable for understanding when patterns were introduced. The clean history was a side effect of AI-assisted development, not deliberate practice.

Pattern 5: The gdbt quickstart Pattern for Onboarding

New engineers should be able to run a single command and have a working local environment. gdbt quickstart became the answer:

  1. Create Python virtual environment
  2. Install dbt and dependencies
  3. Generate .env from template
  4. Clone production Snowflake database to _DEV
  5. Run gdbt compile to verify connection

The key design principle: quickstart should be idempotent. Running it twice should produce the same result as running it once. This lets engineers use it as a "reset to known good state" command, not just a first-run ceremony.

Pattern 6: Incremental Adoption via nv/ Folder

The models/nv/ (non-validated) folder allowed new models to be deployed to production without holding up the migration. A model in nv/ means:

  • It exists and materializes correctly
  • It has NOT been through full data quality validation
  • Downstream consumers should not depend on it yet

This avoided the paradox of "can't ship until validated, can't validate until shipped". Production data is the best validation environment; nv/ made it safe to deploy there.

Pattern 7: Separate "Source Truth" from "Sensor Wiring"

sources_to_dags.yml is the master registry: for each source table, which upstream DAG produces it, what task ID to wait on, what the monitoring type is. This file is the single source of truth that the codegen reads.

Before this file existed, sensor wiring was scattered across hundreds of DAG Python files. A change to an upstream DAG's task ID required hunting through the whole repo. Now it's one file update + gdbt validate.

What Didn't Work (and Was Fixed)

Single monolithic bash script for gdbt — grew unmaintainable. Fixed by extracting into Python modules with clear responsibilities.

Generating sensors at DAG level instead of task group level — caused false dependency signals. An entire DAG would wait for a source that only one of its task groups needed. Fixed by scoping sensors to the task group.

Running full codegen on every gdbt validate — too slow. Fixed by git-diff-aware validation that only regenerates affected files.

Skills that were too prescriptive — early skills specified exact file paths that sometimes didn't exist. Fixed by making skills read-then-adapt: the skill reads the current state of the repo before making decisions, rather than assuming a fixed structure.
