11:Tf90,# What's Next: gdbt as a UV-Installable Python Package

## Current State (as of Mar 2026)

`gdbt` is a bash entrypoint that:
1. Resolves which dbt project to target
2. `cd`s into that project directory
3. Delegates to Python scripts in `scripts/shared/`

The Python modules are solid — `generate_dag.py`, `generate_pipeline.py`, `project_config.py` etc. are clean, well-factored code. The remaining technical debt is the bash layer and the fact that `gdbt` lives inside the repo rather than being a standalone installable tool.

---

## The Target Design

### Distribution
```bash
uv tool install gdbt
# or, for development
uv tool install --editable ./gdbt-package
```

UV's `tool install` creates an isolated environment for the CLI — no virtualenv management, no `pip install`, no `source activate`. It just works.

### Entry Point
A single `pyproject.toml`-based package:
```toml
[project]
name = "gdbt"
dependencies = ["typer", "rich", "dbt-core", "python-dotenv"]

[project.scripts]
gdbt = "gdbt.cli:app"
```

### CLI Structure (Typer)
```python
# gdbt/cli.py
import typer
from gdbt import commands

app = typer.Typer(help="dbt developer CLI for analytics pipelines")

app.add_typer(commands.validate.app, name="validate")
app.add_typer(commands.dag.app, name="dag")
app.add_typer(commands.db.app, name="db")

@app.command()
def run(args: list[str]):
    """Pass-through to dbt with .env loading."""
    ...
```

Each subcommand group gets its own module. Typer handles:
- Auto-generated `--help` at every level
- Type validation on arguments  
- Shell completion generation (`gdbt --install-completion`)

### Rich Output
Replace `print()` with Rich:
```python
from rich.console import Console
from rich.table import Table
from rich.progress import track

console = Console()

# Validation results as a table
table = Table(title="Validation Results")
table.add_column("Model", style="cyan")
table.add_column("Status", style="green")
console.print(table)

# Progress bars for long operations
for model in track(models, description="Regenerating DAGs..."):
    regenerate(model)
```

| Command | Description |
|---------|-------------|
| `gdbt run [args]` | dbt pass-through with .env loading |
| `gdbt validate [--ci] [--force]` | Full validation + codegen |
| `gdbt validate task-groups` | Task group YAML validation only |
| `gdbt dag generate [--all\|<dag_name>]` | Generate Airflow DAG Python files |
| `gdbt dag pipeline` | Interactive pipeline builder |

| `gdbt reset` | Tear down and rebuild local environment |
| `gdbt lint [--fix]` | SQL linting with sqlfluff |

## Cross-Repo Use Case

Once packaged, `gdbt` can be used in any repo that has a dbt project — not just this one. The package discovers dbt projects by looking for `dbt_project.yml` files, and reads `gdbt.config.yml` (or falls back to sensible defaults) for project-specific settings.

This opens the door to:
- Teams in other repos adopting the same YAML-to-DAG codegen pattern
- The skills framework being usable without the repo's `.claude/skills/` directory
- CI workflows consuming `gdbt` as a tool install rather than repo-relative scripts

---

## Migration Path from Current Implementation

1. Extract `scripts/shared/` into `gdbt/` Python package directory
2. Replace bash entrypoint with Typer app
3. Replace print statements with Rich console
4. Write `pyproject.toml` and publish to internal PyPI (or GitHub Packages)
5. Update repo's `scripts/` to install via `uv tool install` in quickstart
6. Keep backward-compatible command surface — all existing `gdbt <command>` invocations continue working

The existing Python module structure maps almost 1:1 to Typer command groups, so the migration is largely mechanical.
12:T1476,# dbt Patterns: Lessons & Reusable Ideas from the Migration

## Pattern 1: CODEGEN Comment Sections as a Contract

Splitting YAML and Python files into "human-written" and "machine-written" sections using `# CODEGEN:` comment markers turned out to be one of the most durable architectural decisions.

- Engineers own everything above the marker
- `gdbt validate` owns everything below it
- Merge conflicts in codegen sections are always resolved by accepting deletions and regenerating
- The `/merge-main` skill handles this automatically

**Why it works:** It eliminates an entire class of merge conflicts. Two engineers can both modify the same task group YAML, and as long as neither touches the CODEGEN section, there's no conflict. The codegen regeneration on the next `gdbt validate` reconciles both changes.

---

## Pattern 2: Encode Review Standards into AI Rules Before Writing Code

The Cursor AI code review rules (`.cursor/rules/`) were written *at the start* of the dbt migration, not after. This meant:

- Every model generated by Cursor was already checked against standards
- Review comments shrunk from "you need to add X" to "looks good, minor nit"
- New team members got standards enforcement before they'd read the docs

The lesson: invest in AI review rules as early infrastructure, not a cleanup step. The ROI compounds because every PR in the project benefits.

---

## Pattern 3: Skills for "Journeys", Not Individual Operations

The initial attempt at skills was one skill per operation: `/analyze-dag`, `/generate-models`, etc. These were useful but required the engineer to orchestrate them manually.

The breakthrough was the **orchestrator skill pattern**: `/create-dbt-migration` composes sub-skills via the Task tool (multi-agent), so engineers get the whole journey in one command. The sub-skills remain individually usable when you need to re-run just one step.

Takeaway: design skills at the level of *what the user wants to accomplish*, then compose smaller skills inside. The user's mental model is "migrate this DAG", not "run analyze, then generate, then validate".

---

## Pattern 4: Git History as a Quality Signal

Because every commit was AI-assisted (either Cursor auto-applying fixes or Claude Code generating migrations), commit quality became unusually high:

- Structured ticket references: `DATA-XXXX: <verb> <thing>`
- Minimal "fix typo" / "oops" commits (the AI caught these before commit)
- Codegen regeneration commits cleanly labeled `Regenerate codegen files after merge from main`
- Work-in-progress commits clearly labeled `WIP:`

This made `git log --grep` searches highly reliable for understanding when patterns were introduced. The clean history was a *side effect* of AI-assisted development, not deliberate practice.

---

## Pattern 5: The `gdbt quickstart` Pattern for Onboarding

New engineers should be able to run a single command and have a working local environment. `gdbt quickstart` became the answer:

1. Create Python virtual environment
2. Install dbt and dependencies
3. Generate `.env` from template
4. Clone production Snowflake database to `_DEV`
5. Run `gdbt compile` to verify connection

The key design principle: **`quickstart` should be idempotent**. Running it twice should produce the same result as running it once. This lets engineers use it as a "reset to known good state" command, not just a first-run ceremony.

---

## Pattern 6: Incremental Adoption via `nv/` Folder

The `models/nv/` (non-validated) folder allowed new models to be deployed to production without holding up the migration. A model in `nv/` means:
- It exists and materializes correctly
- It has NOT been through full data quality validation
- Downstream consumers should not depend on it yet

This avoided the paradox of "can't ship until validated, can't validate until shipped". Production data is the best validation environment; `nv/` made it safe to deploy there.

---

## Pattern 7: Separate "Source Truth" from "Sensor Wiring"

`sources_to_dags.yml` is the master registry: for each source table, which upstream DAG produces it, what task ID to wait on, what the monitoring type is. This file is the single source of truth that the codegen reads.

Before this file existed, sensor wiring was scattered across hundreds of DAG Python files. A change to an upstream DAG's task ID required hunting through the whole repo. Now it's one file update + `gdbt validate`.

---

## What Didn't Work (and Was Fixed)

**Single monolithic bash script for gdbt** — grew unmaintainable. Fixed by extracting into Python modules with clear responsibilities.

**Generating sensors at DAG level instead of task group level** — caused false dependency signals. An entire DAG would wait for a source that only one of its task groups needed. Fixed by scoping sensors to the task group.

**Running full codegen on every `gdbt validate`** — too slow. Fixed by git-diff-aware validation that only regenerates affected files.

**Skills that were too prescriptive** — early skills specified exact file paths that sometimes didn't exist. Fixed by making skills read-then-adapt: the skill reads the current state of the repo before making decisions, rather than assuming a fixed structure.
13:T108a,# dbt Migration: Cutover Pattern & Zero-Downtime Strategy

## The Two-Phase Approach

Migrating data pipelines without downtime requires running old and new in parallel for a validation window. The pattern used here:

**Phase 1 — Build (dbt models created, not yet in production)**
- New dbt models land in `dbt_analytics/models/` 
- Task group YAML config written
- Tables materialized in `_DEV` database during development
- Old Airflow DAG continues running unchanged
- Human review of model logic, schema alignment, data parity

**Phase 2 — Cutover (both pipelines run, then old retired)**
- Cutover DAG runs both old Airflow SQL and new dbt pipeline
- Column mapping validation confirms output equivalence
- After N successful runs in production, old pipeline tasks removed
- Old Airflow DAG archived or paused

---

## Cutover DAG Structure

Generated by the `/create-cutover-dag` skill, the cutover DAG follows a standard pattern:

```python
# Example: ad_data_model -> ads_analytics cutover
dag_id = 'cutover_ad_data_model_to_ads_analytics'

with DAG(...) as dag:
    # 1. Upstream sensors (same as old DAG)
    wait_for_upstream = ExternalTaskSensor(...)
    
    # 2. Run OLD pipeline (existing Airflow SQL)
    old_pipeline = TaskGroup('old_ad_data_model')
    
    # 3. Run NEW dbt pipeline (cosmos DbtTaskGroup)
    new_pipeline = TaskGroup('new_ads_analytics')
    
    # 4. Validation: compare column counts, row counts
    validate_parity = SnowflakeOperator(
        sql="""
        SELECT 
            'old' as source, COUNT(*) as row_count FROM BI_USER.REPORTING.AD_DATA_MODEL WHERE ds = '{{ ds }}'
            UNION ALL
            SELECT 'new', COUNT(*) FROM BI_USER_DEV.REPORTING.ADS_ANALYTICS WHERE ds = '{{ ds }}'
        """
    )
    
    wait_for_upstream >> [old_pipeline, new_pipeline] >> validate_parity
```

The two pipelines run in parallel after their shared upstream sensor, then both feed into a parity validation step. A discrepancy in the validation step fails the DAG and pages the team.

---

## What Gets Validated

The `/validate-tables-dbt` skill performs:

1. **Row count match** — total rows per `ds` partition must be within tolerance
2. **Column presence** — all columns from the old table exist in new table
3. **Null rates** — key non-null columns don't gain nulls
4. **Aggregate sums** — SUM of metric columns matches (within float tolerance)
5. **Distinct value sets** — for dimension columns, same set of values

Snowflake `query_id`s are posted to the PR so reviewers can re-run the comparison themselves.

## Sensor Migration Strategy

A major coordination challenge: old DAGs use task-level `ExternalTaskSensor` pointing at specific Airflow task IDs. New dbt DAGs use `DbtTaskGroup` task IDs generated by Cosmos, which follow a different naming convention.

**Solution:** When an old DAG has sensors pointing at it, the cutover updates the new DAG's task group ID to match the expected external task ID format, or the downstream DAG's sensor `external_task_group_id` is updated to point at the new dbt task group.

This was handled by the sensor override system (DATA-4545) which allows YAML-level overrides without touching the codegen output.

---

## Naming During Transition

Tables that keep their original names for backward compatibility (Looker dashboards, downstream consumers):

- Model file: `int__ad_client_events.sql`
- Materialized as: `BI_USER.REPORTING.AD_CLIENT_EVENTS` (unchanged)
- The `schema` dbt config maps the intermediate model to the right output schema

This "keep the output name" pattern was critical for avoiding a cascade of downstream changes just because the transformation pipeline changed.
14:T18a1,# AI-Assisted Migration: Claude Skills Framework

## Why Skills Instead of Just Prompts

By the time the migration was in full swing (Jan 2026), the team had developed strong institutional knowledge about migration patterns: how to extract DAG structure, which naming conventions to use, how to write cutover DAGs safely, how to validate parity. This knowledge existed in people's heads and in review comments.

**Claude Code skills** codified this knowledge into structured, reusable AI prompts that any team member could invoke. Rather than explaining the entire pattern each time, engineers run `/create-dbt-migration dags/bi_user/ad_events/` and the skill encodes exactly what a senior engineer would know to do.

---

## The Skill Suite

Committed Jan 23, 2026 (DATA-4433), with additions through Feb 2026:

### `/create-dbt-migration` (orchestrator skill)
The top-level entry point for a full Phase 1 migration. Composes the two sub-skills via the Task tool (multi-agent):
1. Launches `/analyze-dag-dbt` as a subagent
2. Waits for the migration context YAML
3. Launches `/generate-models-dbt` as a subagent
4. Returns a review checklist for the human

### `/analyze-dag-dbt`
Reads a DAG directory and produces a structured migration context file at `.dbt-migration/migration_context_<dag_id>.yml`. Extracts:
- All SQL files referenced by `SnowflakeOperator` tasks
- `ExternalTaskSensor` wiring (upstream dependencies)
- `GreatExpectationsOperator` test configurations
- Schedule, SLA, owner metadata
- Target table names and schemas

### `/generate-models-dbt`
Takes the migration context and produces:
- Staging models (`stg_<source>__<table>.sql` + schema YAML)
- Intermediate/legacy models (`int__<name>.sql`)
- Source entries in `sources_to_dags.yml`
- dbt tests converted from Great Expectations expectations
- Task group YAML config for the new pipeline

Enforces all naming conventions automatically — the skill embeds the naming rules so generated code is always correct on first pass.

### `/validate-tables-dbt`
Post-generation validation:
1. `gdbt compile` to check SQL is valid
2. `gdbt run --select <models>` to materialize in dev
3. Row count and column comparison against original Airflow-managed tables
4. Posts Snowflake `query_id`s to the PR for reviewer confirmation

### `/create-cutover-dag`
Creates the zero-downtime cutover DAG that runs the old and new pipelines in parallel:
- Generates the cutover Python DAG from a template
- Adds column mapping validation
- Creates a draft PR automatically
- Posts a PR summary comment with migration context

### `/create-pr`
Creates pull requests with proper JIRA ticket integration, branch naming (`DATA-XXXX-<description>`), and PR body template with migration summary.

### `/create-dbt-migration-batch`
Runs multiple migrations in parallel using Claude's multi-agent Task tool:
```
/create-dbt-migration-batch dags/bi_user/ad_events/ dags/grindr_analytics/dim_profile/
```
Launches one agent per DAG simultaneously, then aggregates results.

### `/deprecated-bi-user-tables`
Reference skill for migrating deprecated `BI_USER` source tables to the new intermediate profile attribute models. Encodes the specific column mappings and semantic equivalences.

### `/merge-main`
Merges `main` into the current feature branch, auto-resolves conflicts in codegen files (since they're always regenerated), and runs `gdbt validate` to rebuild them cleanly.

## How AI Was Used to *Build* the Skills

The skills themselves were built with AI assistance — a meta pattern. The workflow:

1. A senior engineer performs a migration manually once, carefully
2. Claude Code (via Cursor) observes the pattern across the PR diff
3. The engineer describes the pattern to Claude: "that's always how cutover DAGs should look"
4. Claude drafts the skill YAML/Markdown
5. The engineer refines it on 2–3 real migration runs
6. The skill is committed to the repo and available to the whole team

The Cursor AI code review rules (`.cursor/rules/airflow-code-review.mdc`, `.cursor/rules/sql-code-review.mdc`) were built the same way — patterns extracted from actual review comments, formatted as machine-readable rules that Cursor applies automatically during review.

**Key insight:** The AI wasn't used to replace engineering judgment. It was used to *encode* engineering judgment so it could be applied consistently at scale.

## Multi-Agent Migration Workflow

The most powerful pattern: running migrations in parallel across multiple DAGs simultaneously. Example from the advertising cluster migration (DATA-4531):

```
/create-dbt-migration-batch \
  dags/bi_user/ad_data_model/ \
  dags/bi_user/ad_events/ \
  dags/bi_user/ad_revenue_rolling_avg/
```

This launches three independent agents. Each one:
- Reads the DAG
- Generates models
- Creates task group YAMLs
- Writes to non-overlapping paths in the repo

The main agent merges results and creates one PR. Work that would take 3–4 days manually was completed in a single afternoon.

## Cursor vs. Claude Code: Complementary Roles

Both tools were used, with distinct roles:

**Cursor (via `.cursor/rules/`):**
- In-editor AI review during development
- Applies code review findings automatically with search_replace
- Always-on linting and standard enforcement
- Used for exploratory SQL writing and model iteration

**Claude Code (via `.claude/skills/`):**
- Structured, multi-step workflow automation
- Cross-file operations (read DAG → write multiple dbt files → update YAML configs)
- PR creation and git operations
- The "do the whole thing" command

The combination meant engineers could write a model interactively in Cursor, then hand off the migration scaffolding to Claude Code skills.

15:T1696,# Pipeline Framework: YAML-to-Airflow Codegen

## Core Idea

Engineers should never need to write Airflow Python to add a new dbt pipeline. They configure what runs in YAML files inside the dbt project, run `gdbt validate`, and the framework generates correct, production-grade Airflow DAG Python files.

This decoupling means:
- Data engineers work entirely in the dbt repo
- DAG Python files are treated as build artifacts, not source code
- Sensor wiring, task groups, and scheduling are derived from the dbt model graph

---

## Config Hierarchy

```
dbt_analytics/pipelines/
├── config.yaml                    # Project-level settings
├── dags/                          # One YAML per Airflow DAG
│   ├── profile_engagement.yml
│   ├── product_analytics.yml
│   └── ...
└── task_groups/                   # One YAML per logical task group
    ├── intermediate/
    │   ├── profile_engagement.yml
    │   ├── product.yml
    │   └── profile_attributes/
    │       ├── profile_device.yml
    │       └── ...
    ├── mart/
    └── staging/
```

### DAG Config (`dags/<name>.yml`)

```yaml
name: profile_engagement
description: Daily pipeline for profile engagement models.

output_dag: dags/dbt/analytics/profile_engagement.py
schedule: daily 18:00
start_date: 2026-03-08
catchup: true
depends_on_past: true

config:
  dbt_project: analytics
  warehouse:
    local: DE_ETL_WH
    prod: DE_ETL_WH

task_groups:
- file: intermediate/profile_engagement.yml
- file: mart/profile_engagement.yml

# CODEGEN sections below — do not edit manually
```

### Task Group Config (`task_groups/<layer>/<name>.yml`)

```yaml
name: profile_engagement
dbt:
  selector: path:models/intermediate/profile_engagement

# CODEGEN: Auto-populated by gdbt validate
included_models:
- key: int__profile_taps_daily
  type: incremental
  target: BI_USER.REPORTING.PROFILE_TAPS_DAILY

upstream_task_groups:
- intermediate/product

sources:
- bi_user.client_event_daily
- eventbus.login_event
```

## Codegen Process

When `gdbt validate` runs, the Python codegen layer (`generate_dag.py`, `generate_pipeline.py`) does:

1. **Parse task group YAMLs** — load `dbt` selectors, extract included models by running `dbt ls`
2. **Resolve upstream dependencies** — for each source in a task group, look up `sources_to_dags.yml` to find the upstream DAG + task ID
3. **Compute execution deltas** — given the downstream DAG schedule and upstream DAG schedule, calculate the `timedelta` for each `ExternalTaskSensor`
4. **Populate CODEGEN sections** — write auto-discovered sensors, upstream DAGs, downstream DAGs, and stats back into the YAML files with `# CODEGEN:` comment markers
5. **Generate Airflow Python DAG** — render `dag_template.jinja2` with all config, producing a fully valid Airflow Python file
6. **Validate references** — check that all referenced task group files exist, all sensor external_dag_ids exist

The `# CODEGEN:` comment is the contract: anything below it in a YAML or Python file is machine-written and will be overwritten on the next `gdbt validate --fix`.

---

## Generated DAG Structure

The generated Python DAG always follows the same structure:

```python
# CODEGEN: This file is code-generated and should not be modified manually
# Generated from: dbt_analytics/pipelines/dags/profile_engagement.yml
# To regenerate: cd dbt_analytics && gdbt validate

from cosmos import DbtTaskGroup, ProjectConfig, RenderConfig
from plugins.util import pipeline_builder as pb

SNOWFLAKE_WAREHOUSE = pb.utils.get_warehouse_for_env(local_warehouse='DE_ETL_WH', prod_warehouse='DE_ETL_WH')

dag_config = pb.config.DagConfig(dag_id='profile_engagement', ...)

with DAG(**dag_config.to_dag_kwargs()) as dag:
    # For each task group:
    with TaskGroup('intermediate_profile_engagement') as tg:
        with TaskGroup('source_sensors') as _sensors:
            ExternalTaskSensor(task_id='wait_for_...', ...)
        
        DbtTaskGroup(
            group_id='dbt_models',
            project_config=ProjectConfig(...),
            render_config=RenderConfig(select=['path:models/intermediate/profile_engagement'], ...),
            ...
        )
```

**Astronomer Cosmos** (`DbtTaskGroup`) handles the actual dbt model execution within Airflow, providing per-model task visibility in the Airflow UI.

## Sensor Framework

The sensor system evolved significantly:

**v1 (manual):** Engineers wrote `ExternalTaskSensor` directly in DAG Python files

**v2 (sources_to_dags.yml):** Sources declared in YAML; sensors auto-generated from the file

**v3 (task group sensors):** Sensors scoped to the task group that needs them, not the DAG level

**v4 (sensor overrides, DATA-4545):** Per-task-group `sensor_config` in YAML allows overriding sensor type, retries, and execution delta without touching codegen:

```yaml
# In a task group YAML, under sensor_config
sensor_config:
  freshness_check_override:
    source: chat_sensitive.chat_platform
    table: CHAT_SENSITIVE.REPORTING.CHAT_PLATFORM
    monitor_type: hourly
  external_task_sensor_override:
    external_dag_id: some_other_dag
```

## Incremental Patterns

The `delete+insert` strategy became the standard for all incremental models in `dbt_analytics`:

```sql
{{
    config(
        materialized='incremental',
        incremental_strategy='delete+insert',
        unique_key=['ds'],           -- or ['ds', 'hr'] for hourly
        on_schema_change='append_new_columns'
    )
}}
```

**Why `delete+insert` over `merge`:**
- Snowflake `MERGE` locks tables and is slow at scale
- `delete+insert` is faster and aligns with the daily partition model
- `on_schema_change='append_new_columns'` allows schema evolution without full rebuilds

**Hourly models** use `['ds', 'hr']` as the composite unique key and inject both `ds()` and `hr()` Jinja macros for partition filtering.

## Background

The data platform originally ran entirely on Airflow DAGs: SQL files living next to Python orchestration files, table creation handled imperatively via `SnowflakeOperator` tasks, and pipeline dependencies wired manually through `ExternalTaskSensor` chains. Over time this created:

- **Maintenance drag** — SQL changes required touching both the query file and the DAG wiring
- **No lineage** — no automatic tracking of which tables depended on which sources
- **Inconsistent testing** — Great Expectations tests were bolted on per-DAG, not part of the model contract
- **Hard to onboard** — new engineers had to understand both Airflow and bespoke SQL patterns to contribute

The migration to dbt solved all of these by decoupling *transformation logic* (dbt) from *orchestration* (Airflow), while building a codegen layer so users could configure pipelines entirely from YAML in the dbt repo.

19:T3579,# HotLap: Complete Metric Logic — How Every Score Is Computed

Source files: `ml/src/iracing/process_session.py` (per-lap analysis), `ml/src/iracing/queries/session_analysis.py` (session aggregation), `ml/src/iracing/queries/progressive_guide.py` (multi-session learning).

## Raw Telemetry Inputs

iRacing captures at 60 Hz. Each lap row in `sim_racing.telemetry` has:
- `throttle` (0-1), `brake` (0-1), `steering_angle` (degrees), `speed` (kph)
- `lap_dist` (meters from start/finish), `gear` (int)
- `lat_accel`, `long_accel` (m/s²) — needed for lateral G shape scoring

`dt = 1/60 ≈ 0.0167s` per frame. All time-based thresholds are frame-count × dt.

---

## Step 1: Corner Detection

### Braking Zone Detection (`find_braking_zones`)
A braking zone is any contiguous run where `brake > 0.1` for more than 5 frames (~83ms). Returns `(start_idx, end_idx)` pairs.

### Chicane / S-Curve Splitting (`_split_zones_by_steering`)
For each braking zone, check if steering crosses zero with significant magnitude on both sides (both > 15°). If so, the zone is a chicane — split it at the zero-crossing point. Both sub-zones must be > 5 frames to be kept as separate corners.

**Why:** A chicane produces one brake application but two distinct driving actions. Scoring them together would average out and hide technique on each direction.

### Liftoff Corner Detection (`_find_liftoff_zones`)
High-speed corners where the driver lifts off throttle but doesn't brake. Detection criteria:
1. Throttle drops from > 0.7 to < 0.3 within 1 second
2. Entry speed > 80 kph
3. Max steering angle in zone > 10°
4. Max brake < 0.1 (not a braking corner)
5. Speed drops by > 5% from entry (actually slowing down)
6. Doesn't overlap with any existing braking zone ± ~1 second

Liftoff corners get line quality scoring but NOT trail brake scoring (there's no brake to taper).

### Corner Filtering (false positive removal)
After detection, each zone is filtered before being added to results:
- **Spin/stop filter:** `min_speed < 20 kph` → skip (car nearly stopped)
- **Brake tap filter:** `min_speed > entry_speed × 0.9` → skip (car barely slowed, not a real corner)

---

## Step 2: Per-Corner Basic Metrics

For each detected corner zone `[zone_start, zone_end]`, a `pickup_window` is opened: `zone_end + 3s` of frames ahead (to catch throttle application).

`pickup_idx` = first frame after zone_end where throttle > 0.3.

**`coasting_time`**: sum of frames where `throttle < 0.05 AND brake < 0.05` from zone_start to pickup. Multiplied by dt → seconds.

**`brake_point_dist`**: `lap_dist[zone_start]` in meters from start/finish.

**`min_speed_kph`**: `min(speed[zone_start:zone_end])`.

**`exit_speed_kph`**: speed at `zone_end + pickup_idx`.

**`brake_attack_rate`**: time (seconds) from brake onset to reaching 90% of peak brake pressure.

**`speed_ratio`**: `min_speed / entry_speed` — raw fraction, not scored into 0-100.

---

## Step 3: Trail Brake Score (0-100 composite)

Weights: `brake_taper 35%`, `connection 35%`, `rotation_readiness 30%`

### Component A: Brake Taper (`_score_brake_taper`)
Measures smoothness of brake release — the gradual progressive trailing off of brake pressure through the corner.

**Algorithm:**
1. Find peak brake pressure index in zone
2. Find where brake drops below 5% after peak (`taper` segment)
3. Build ideal = linear ramp from `peak_val` to 0 across same length
4. RMSE of actual taper vs ideal: `rmse = sqrt(mean((taper - ideal)²))`
5. Score = `(1 - rmse / peak_val) × 100` → 100 means perfect linear ramp, 0 means instant dump
6. **Snap penalty:** if `taper_duration < 0.3s`, score × `(duration / 0.3)` — a real taper needs time

Score interpretation: 100 = perfectly smooth linear release over >0.3s. 0 = brake dumped instantly.

### Component B: Connection (`_score_connection`)
Measures quality of the brake-to-throttle handoff — the critical transition between pedals.

Combines two previously separate metrics (coasting gap + overlap quality) into one.

**Coasting penalty (60% weight):** Piecewise linear on `coasting_time`:
```
0.0s → 100
0.1s → 80
0.3s → 50
0.6s → 20
1.0s → 0
```

**Overlap quality (40% weight):**
- No overlap + coasting ≤ 0.05s → 90 (clean instant handoff, great)
- No overlap + coasting > 0.05s → 50 (full gap, coast score handles penalty)
- With overlap: scored on duration + brake lightness
  - Duration: ≤0.2s → 100; 0.2-0.4s → 100→70; >0.4s → 70→30
  - Lightness: avg brake during overlap; <10% → 100; >40% → 20 (riding the pedals)
  - `overlap_quality = 0.6 × duration_quality + 0.4 × lightness`

**Final: `0.6 × coast_score + 0.4 × overlap_quality`**

Score interpretation: 100 = zero coasting + brief light overlap. The sweet spot is zero dead pedal with ~0.1s of light brake as throttle comes in.

### Component C: Rotation Readiness (`_score_rotation_readiness`)
Measures whether the car is rotated (pointed toward corner exit) before the driver applies throttle. Penalizes applying gas while still turning in.

**Sub-score A: Steering unwind at throttle onset (70% of component)**

`thr_idx` = first frame after zone_end where throttle > 10%.

`unwind_frac = 1 - (steering_at_thr_idx / peak_steering)` — what fraction of the peak steering angle has been unwound when throttle comes on. 0 = still at peak (no rotation), 1 = fully straight.

Scored via piecewise breakpoints — different for heavy vs light corners:
- **Heavy corner (peak steering > 30°):**
  ```
  0.0 → 0
  0.15 → 30
  0.30 → 60
  0.50 → 85
  0.70 → 100
  ```
- **Light corner (peak steering ≤ 30°, kinks/sweepers):**
  ```
  0.0 → 30   (leniency — light corners need less unwind)
  0.10 → 60
  0.20 → 85
  0.35 → 100
  ```

Steering derivative bonus/penalty: if steering is still increasing at throttle onset → -15; if unwinding → +10.

**Sub-score B: Throttle reapplications (30% of component)**
Counts throttle lift-reapply events (>10% → <5% → >10%) after brake release:
```
0 reapplications → 100
1 → 55
2 → 20
3+ → 0
```

**Final: `0.70 × unwind_score + 0.30 × reapp_score`**

---

## Step 4: Line Quality Score (0-100 composite)

Weights: `steering_corrections 35%`, `steering_smoothness 35%`, `lateral_g_shape 30%`

Window: expanded to `zone_start` through `pickup_end + 1s` (covers full corner including exit).

### Component A: Steering Corrections
Counts mid-corner steering reversals — real direction changes indicating the driver is fighting the car.

**Algorithm:**
1. Low-pass filter steering with 200ms moving average (~12 frames at 60Hz) to remove force feedback noise
2. Compute filtered steering rate (derivative)
3. Threshold: `max(8, min(30, 2000/avg_speed))` degrees/frame — higher speed = slightly more tolerance
4. Count zero-crossings in rate signal (sign changes above threshold)

Piecewise scoring:
```
0 → 100
1 → 90
3 → 70
5 → 50
7 → 30
10 → 10
14 → 0
```

### Component B: Steering Smoothness
Measures jerkiness of steering inputs, normalized by what's physically "normal" for this corner.

`steer_rate_std = std(derivative of filtered steering)`

Three normalization factors:
- **Speed factor:** `(avg_speed / 100)^0.5` — sqrt scaling, faster corners naturally have more rate
- **Range factor:** `1 + min(steer_range / 80, 0.5)` — wider steering arc tolerates more rate (cap at 1.5x)
- **Brake factor:** `1 + avg_brake × 0.3` — heavy braking unsettles the car (cap at ~1.24x)

`normalized_std = steer_rate_std × speed_factor / (range_factor × brake_factor)`

Piecewise scoring:
```
0 → 100
25 → 95
45 → 80
65 → 60
85 → 40
110 → 20
140 → 0
```

### Component C: Lateral G Shape
A committed, correct line produces a single smooth peak in lateral acceleration. Multiple peaks indicate the driver adjusted mid-corner (changed line, corrected).

**Algorithm:**
1. Light smoothing of lateral G (kernel size = `max(3, min(7, window_length/20))`)
2. Count distinct peaks: regions where lateral G exceeds 40% of max G
3. Count complete above-threshold cycles (below→above→below transitions)
4. Score: 1 peak → 100, 2 → 60, 3 → 20, 4+ → 0

Falls back to 50 if not enough G force to judge (< 0.1 G max), or if lat_accel data unavailable.

---

## Step 5: Session-Level Aggregation (`query_session_analysis`)

### Warm-Up Exclusion
First 2 laps per sub-session tagged as `is_warmup = True`. Cold tires + driver not yet in rhythm. All aggregate statistics are computed from "flying laps" only (non-warmup). If < 2 flying laps exist, fall back to all laps.

### Consistency Metrics
From flying laps:
- `lap_time_std`, `lap_time_mean`, `lap_time_best`, `lap_time_worst`
- **Theoretical best:** min sector time per sector summed across flying laps
- **Sector variance:** std per sector across flying laps
- **Brake point variance:** std of brake point distances per corner cluster (in meters)
- **Min speed variance:** std of corner min speeds per cluster

### Corner Clustering (`_cluster_corners`)
Groups corner observations from all flying laps by distance. Two observations cluster together if their brake point distances are within 80m. Used for per-corner consistency stats across laps within a single session.

### Orphan Cluster Merging (session-level)
After matching clusters to named guide corners, any unmatched ("orphan") cluster within 150m of a matched cluster gets merged into it. Prevents split braking zones from appearing as phantom `C4`, `C6` extra corners.

### Best vs Average
Compares best lap vs session average on: throttle_mean, full_throttle_pct, coasting_pct, steering_smoothness, throttle_smoothness.

### Race Analysis (when session_type = "race")
Clean air vs traffic technique comparison: separate averages of coasting_pct and steering_smoothness for laps tagged as `clean` vs `traffic` or `heavy_traffic`. Traffic proximity from `proximity.close_ahead_pct` in lap_analysis.

## Step 6: Progressive Guide — Multi-Session Learning

The progressive guide accumulates corner data across all sessions for a car/track combo. Each corner observation is evaluated independently — a lap where the driver coasts through T1 contributes nothing to T1's reference but may contribute high-quality data for T5.

### Phase 1: Collect All Observations
Query all laps for the car/track combo. For each lap's `lap_analysis->'corners'`, extract: `dist_m` (brake_point_dist), `min_speed_kph`, `trail_brake_score`, `coasting_time`, `line_quality_score`, `gear_at_apex`, `lap_time`, `session_id`.

### Phase 2: Cluster by Distance
Sort all observations by distance, then sweep: each new observation either joins the current cluster (within 80m of cluster center) or starts a new one. Cluster center updates as running mean.

### Phase 3: Outlier Removal (IQR fence)
For each cluster's speed distribution:
```python
q1, q3 = percentile(speeds, [25, 75])
iqr = q3 - q1
upper_fence = q3 + 2.0 * iqr  if iqr > 5  else  q3 + 30
lower_fence = max(1, q1 - 2.0 * iqr)  if iqr > 5  else  max(1, q1 - 30)
```
Observations outside this fence are data errors (mis-clustered zones, off-track events). If IQR is very small (<5 kph), use flat ±30 kph fence instead.

### Phase 4: Effort Filter (P30 threshold)
After outlier removal, discard observations below the 30th percentile of speed for that corner. These are "coasting" laps — the driver wasn't trying hard at this corner. Reference values should only reflect genuine attempts.

### Phase 5: Reference Speed (CV-gated)
`cv = std(high_effort_speeds) / mean(high_effort_speeds)` — coefficient of variation.

| CV | Interpretation | Reference Speed |
|---|---|---|
| < 5% | Settled technique — consistent | P80 of high-effort speeds |
| < 10% | Developing | P70 |
| ≥ 10% | Experimenting | Median (conservative) |

**Why percentiles, not max:** The driver may occasionally fluke a fast corner entry that they can't reproduce. The guide tracks what they can *consistently* achieve.

### Phase 6: Confidence Rating
```
high:   CV < 5%  AND unique_sessions ≥ 3  AND high_effort_count ≥ 15
medium: CV < 10% AND unique_sessions ≥ 2  AND high_effort_count ≥ 5
low:    everything else
```

Three confidence gates also map to the coaching "skill stage" cold-start: learning (<3 sessions), developing (3-5), stable (5+).

### Phase 7: Overdriving Detection
For each corner in the guide:
```
overdriving = (P50_speed > ref_min_speed) AND (tb_mean < 50 OR lq_mean < 50)
```
The driver is going faster than their reference speed, but their technique scores are degraded — they're beyond their limit.

### Corner Naming (Lovely Track Data)
Two-step fuzzy match:
1. Find the closest matching track in `sim_racing.lovely_tracks` by name similarity (pg_trgm, threshold > 0.3), then pick closest by track length (using actual telemetry max `lap_dist` as length)
2. For each cluster, find which Lovely turn's distance range it falls in → assign the human name ("Esses", "Horseshoe", etc.)

1b:T19f1,# HotLap: How AI Is Used and What Was Learned

## Two Distinct AI Roles

### 1. Claude as the Coaching Engine (product feature)
Claude reads structured telemetry data and writes driving coaching briefs. This is an end-user facing feature.

### 2. Claude Code as the Dev Tool (developer tool)
The entire app was built by a single developer using Claude Code, with skills defined in `.claude/skills/`. The coaching skill, the progressive guide system, the metric decomposition — all of it was designed and implemented through AI-assisted development.

---

## How the Coaching Pipeline Works

### Data Flow
```
telemetry (.db SQLite) 
  → ingest_capture.py (Python) 
  → session_analysis.py (per-lap corner scores) 
  → progressive_guide.py (aggregated reference values per corner) 
  → gather-coaching-context (collects everything into one JSON blob) 
  → Claude (writes coaching brief) 
  → saved to DB + shown in web UI
```

### What Claude Receives
The coaching context JSON includes (~12K chars after slimming from 70K):
- Session metadata (car, track, date, num_drivers, session types)
- Per-lap corner analysis: trail_brake_score, line_quality_score, min_speed_kph, gear, coasting_time, speed_ratio
- Per-component breakdowns: brake_taper, connection, rotation_readiness / lateral_g_shape, steering_smoothness, steering_corrections
- Progressive guide reference values (what good looks like for this driver at this track)
- Prior coaching at same track (most recent session)
- Prior coaching cross-track (if different track, for transfer tests)
- Fundamentals (car-level scores, focus areas, history)
- Technique wiki (relevant markdown files)

### Model Choice
- Production: `claude-sonnet-4-6` (v2 "open coach" style)
- Tested: `claude-haiku-4-5` for cost reduction — fast but coaching quality dropped noticeably on complex sessions
- v1 was a "prescriptive" style (more rigid, step-by-step instructions); v2 is an "open coach" (more contextual, adapts to what the driver actually needs)

---

## Key Learnings

### 1. Aggregates are useless for coaching — decompose everything
The first version used `trail_braked_pct` (binary: did the driver coast for less than 0.15s?). The driver repeatedly "failed" trail braking without understanding why. The fix: decompose into 3 sub-components (brake_taper, connection, rotation_readiness), score each 0-100, surface the weakest one. The driver immediately understood: "my taper is 74 but connection is 26 — I'm lifting the brake before touching the throttle."

**Pattern to reuse:** Any metric where the coaching says "improve X" but the driver doesn't know how → decompose into sub-skills, score independently, color-code.

### 2. Token reduction requires curation, not summarization
Initial coaching context was 70K+ chars (raw telemetry dumps). Slimmed to ~12K by:
- Only sending aggregated corner metrics, not raw telemetry arrays
- Limiting prior coaching to the most recent session (not the full history)
- Pre-computing all derived metrics in Python, not leaving calculation to the LLM

**Don't ask Claude to compute from raw data.** Pre-aggregate in Python. Claude's job is synthesis and communication, not math.

### 3. Prompt versioning matters when you change philosophy
Added a `coaching_version` field to distinguish v1 (prescriptive) vs v2 (open coach) prompts. When reviewing progress across sessions, the prior coaching style affects how you interpret what advice was given. Tag your prompts.

### 4. The coaching *chain* is as important as per-session analysis
Single-session coaching is shallow. The most valuable coaching pattern is the **transfer test**: when the driver moves to a new track, compare their fundamentals (coasting %, brake smoothness, steering corrections) against the last session at the previous track. Did the skill transfer or did it reset?

Cross-track coaching also enables catch-up: "Last week at Summit Point you were working on brake-throttle connection. Your first session at Oschersleben shows connection scores of 44 — better than your Summit baseline of 26. That transferred."

### 5. Horizontal coaching before vertical — AI needs explicit instructions for this
Without being explicitly told, Claude will drill into the worst corner in detail. But for beginners, this is wrong: if 3 other corners have worse consistency issues, fixing corner 8's overlap first is wasted effort. The coaching skill prompt explicitly instructs "scan all corners first, get everything to adequate, then drill." This is not intuitive for LLMs without the instruction.

### 6. Mental bandwidth model: why skill gates exist
The prompt explains to Claude *why* horizontal-first coaching works: a beginner's brain is fully occupied with vision, racing line, and braking fundamentals. There is literally no spare mental bandwidth to also process trail braking technique. The coaching should tell the driver: "Make [X] automatic before adding [Y] — not because you're not allowed to, but because your brain can't process both yet." This framing gets better driver buy-in than "you're not at that skill level yet."

### 7. Live overlay coaching requires different feedback than post-session coaching
The sim capture tool shows per-corner feedback in real-time (trail brake score, speed delta vs guide, weakest component). This feedback must be:
- **Immediate** (shown within 2-3 seconds of corner exit)
- **Single metric** (the driver can't read a paragraph mid-session)
- **Actionable in the next lap** (not a historical trend)

Post-session Claude coaching is the opposite: narrative, trend-based, multi-lap patterns.

### 8. Debug logging enables AI-to-AI comparison
The `live_log` table captures exactly what the driver saw during the session (banners, corner scores, next-corner previews). This means you can compare live feedback (what Claude's coaching said to focus on) against post-session offline analysis (what actually happened). Discovered one case where live scores were systematically lower than offline scores due to smoothing kernel difference — only visible because both sides were logged.

---

## What Didn't Work

- **Static track guides:** Required pre-built reference data per car/track combo. Never scaled. Progressive guide replaced them entirely.
- **`race_events` table:** Awkward date-based grouping. Uploads are a better container. Slated for removal.
- **Letting Python scripts mutate the `uploads` table:** Node owns that table now. Mixing ownership caused status-update races.
- **Token-heavy contexts:** 70K chars caused slow responses and increased cost. Aggressive pre-aggregation in Python is required.
1c:T1372,# HotLap: iRacing Coaching App — Product Overview and Evolution

## What It Is

HotLap is a personal iRacing coaching app built for one driver (the developer, a beginner). It closes the loop between sim racing sessions and structured skill improvement:

1. A **Windows capture tool** (PyInstaller .exe) runs alongside iRacing, captures telemetry to a SQLite `.db` file, and shows a real-time overlay with per-corner feedback.
2. After a session, the capture tool uploads the `.db` to GCS.
3. A **Next.js web app** (Cloud Run) ingests the file via Python ML scripts, runs analysis, and stores structured results in PostgreSQL (Neon).
4. The web app shows a **progressive guide** — a per-corner breakdown of the driver's technique built from their own telemetry across sessions.
5. A **Claude-powered coaching skill** reads all that data and writes a structured coaching brief.

**Stack:** Next.js 16 (App Router, React 19) + Python ML pipeline. Node spawns Python scripts via `child_process.spawn`. DB is Neon PostgreSQL with Drizzle ORM.

- Second most-practiced: **Okayama International Circuit** — 22 sessions, 110 laps

### Week 2 (March 8-16): Architecture Shift — Progressive Guide

The biggest architectural change: **replaced static track guides with a progressive guide** built from the driver's own telemetry.

**Why:** Static guides require pre-built reference data. Progressive guides learn from the driver's actual observations across sessions — no cold-start dependency, and the reference speeds reflect what *this driver in this car* can achieve, not a generic benchmark.

**Key design:** Each corner observation is evaluated independently. A lap where the driver practices T6-T7-T8 and coasts through T1 contributes high-quality data for T6-T7-T8 and nothing for T1. The guide learns from each corner's best observations, not from "best laps."

Confidence gates: learning (<3 sessions), developing (3-5 sessions), stable (5+).

### March 9-10: Metric Decomposition

Trail brake metric evolved from binary (`trail_braked_pct`) → composite 0-100 score with 3 sub-components:
- **brake_taper** (35%): smoothness of brake release through the corner
- **connection** (35%): no dead pedal gap + brief light overlap
- **rotation_readiness** (30%): whether the car is rotated before throttle application

Line quality similarly decomposed into: `lateral_g_shape`, `steering_smoothness`, `steering_corrections`.

This was the most impactful coaching improvement — instead of "your trail braking is weak," the coach could say "your taper is fine at 74 but connection is 26 — you're lifting the brake before getting on throttle."

### March 15-16: New Tracks + Deprecating Fundamentals

Started practicing Motorsport Arena Oschersleben (6 sessions). Cross-track coaching chain established — prior coaching from Summit Point carries into new track sessions as a "transfer test."

Fundamentals system being deprecated in favor of inline skill stage computed from progressive guide data.

## Core Coaching Philosophy (Documented in CLAUDE.md)

Two key principles emerged from real usage and feedback:

1. **Decompose metrics into actionable components.** Binary pass/fail metrics hide which sub-skill is lacking. Every aggregate metric should become a composite score with independently-scored components, color-coded (green ≥70 / yellow ≥40 / red <40), with the weakest component surfaced.

2. **Horizontal first, then drill.** Scan ALL corners before going deep on any one. A driver with inconsistent brake points across 4 corners should not be coached on brake-throttle overlap at corner 8. Get everything to "adequate for your level" first.

Both principles were discovered through real frustration — the driver not understanding coaching feedback and not improving — not from upfront design.
1d:T101a,# AI Workflow: Hooks, Safety Guards, and Infrastructure Scripts

## Philosophy

The workflow is designed so that the "easy path" is also the safe path. Rather than relying on prompts to prevent mistakes, safety is enforced in scripts that run before/after tool calls.

## Hooks (Pre/Post Tool Execution)

### `protect-main-on-pr.sh` (Pre-tool hook on `gh pr create`)

**Problem it solves:** Developers sometimes accumulate commits on `main` locally (especially when working quickly), then try to open a PR from `main` — which targets `main` as both source and base.

**What it does:**
1. Detects if the current branch is `main` with unpushed commits
2. Extracts a branch name from the most recent commit message
3. Automatically creates a new branch from those commits
4. Moves the commits to the new branch
5. Allows the `gh pr create` to proceed normally

This runs **before** every `gh pr create` call, so neither Claude nor developers can accidentally create PRs from main.

### `capture-issue-create.sh` (Post-tool hook on `gh issue create`)

**Problem it solves:** Issues are created without being reviewed, leading to incomplete or ambiguous issues that waste triage time.

**What it does:**
1. Parses the successful `gh issue create` output to extract the new issue number
2. Outputs a reminder: "Issue #N created. Don't forget to run `@issue-reviewer #N` to validate."

This creates a nudge without making review mandatory (the decision remains with the human).

## Sandboxed GitHub CLI: `gh.sh`

A wrapper around `gh` that enforces a command allowlist. Skills that only need to read use `gh.sh` instead of raw `gh`.

**Allowed operations:**
```
gh issue view
gh issue list
gh issue search
gh issue label list
gh pr view
gh pr diff
gh pr list
```

**Blocked operations:** anything that writes — `gh issue create`, `gh issue close`, `gh issue comment`, `gh pr create`, `gh pr merge`, `gh pr review`, branch operations.

The script validates both the command and its flags before passing through to `gh`.

## `edit-issue-labels.sh`

Manages issue labels safely:
```bash
edit-issue-labels.sh --issue 156 --add-label stage:triaged --remove-label stage:backlog
```

**Safety features:**
1. Fetches the valid label list from GitHub before applying
2. Validates every label in `--add-label` against the valid list
3. Reports what was added/removed (doesn't silently fail on invalid labels)

Prevents skill prompts from accidentally applying non-existent labels (which would silently do nothing in raw `gh`).

## `settings.json` Permission Configuration

The Claude Code settings file defines what Claude can and cannot do autonomously:

```json
{
  "permissions": {
    "allow": [
      "Bash(go test:*)",
      "Bash(go build:*)",
      "Bash(gh issue view:*)",
      "Bash(gh pr view:*)"
    ],
    "ask": [
      "Bash(git merge:*)",
      "Bash(git push --force:*)",
      "Bash(gh pr merge:*)"
    ]
  },
  "hooks": {
    "PreToolUse": [".claude/hooks/protect-main-on-pr.sh"],
    "PostToolUse": [".claude/hooks/capture-issue-create.sh"]
  }
}
```

Destructive git operations are in `"ask"` — Claude must get explicit confirmation before running them, even in automated contexts.

1f:T1138,# Skill System Architecture: File-Based Prompts with a Unified Runner

## Skills Are Files

Each skill is a `SKILL.md` file stored in `.claude/skills/<name>/SKILL.md`. The file has YAML frontmatter followed by the full prompt:

```yaml
---
name: triage
description: Analyze a GitHub issue and propose an implementation approach
allowed-tools: Read, Glob, Grep, Bash(gh:*), Bash(go:list)
---

You are a senior engineer on the Cronos project...
[full prompt follows]
```

The skill file IS the prompt — loaded verbatim at runtime. This makes skills:
- Human-readable (anyone can review what Claude will do)
- Version-controlled (skill evolution is tracked in git history)
- Editable without code changes (no redeployment needed)

## The Unified Runner: `run-skill.sh`

A single script handles local and CI invocation identically:

```bash
.claude/scripts/run-skill.sh /triage 156
.claude/scripts/run-skill.sh /implement 156
.claude/scripts/run-skill.sh /audit 156
```

### What the Runner Does

1. Parses skill name and issue/PR number from args
2. Loads the SKILL.md file from `.claude/skills/<name>/`
3. Determines tool allowlist (READ_TOOLS vs WRITE_TOOLS per skill)
4. Sets `--max-turns` per skill (ask: 10, triage: 20, audit: 30, implement: 40)

### The --print Architecture (Critical Bug Fix)

Early versions gave all skills access to `gh issue comment`. This caused a critical bug: triage/audit would accidentally post raw file contents as issue comments (e.g., the full `model_app.go` file appearing as a comment).

Root cause: when Claude had `gh issue comment` access during read operations, it would call it at unexpected times — sometimes with tool output rather than final analysis.

Fix: read-only skills run with `--print` (which outputs final response to stdout and exits). The shell script captures stdout and posts it as the comment. Claude never calls `gh issue comment` directly in read-only mode.

Write skills (`/implement`) still have `gh issue comment` access because they need to post the implementation log mid-run (before the PR exists).

21:Tc5f,# Audit Skill: Deep Exploratory Analysis for Complex Issues

## When to Use

Run `/audit` instead of `/triage` when:
- Triage returned 🔴 Red
- The issue explicitly touches architecture (models, state machines, accounting, multi-tenancy)
- The scope is unclear and may be too large for a single implementation
- You want trade-offs and options, not just a prescribed approach

## What It Does

Audit performs a structured multi-dimensional analysis:

### Seven Audit Dimensions

1. **Data Model Audit** — All relevant models, relationships, GORM hooks. What's stored, what's derived.
2. **Business Logic Audit** — Full code path trace from handler to DB. Error handling, side effects, edge cases.
3. **State Machine Audit** — All valid transitions, gaps, edge cases. Is every state reachable? Every exit valid?
4. **Accounting Audit** — Double-entry balance invariants, journal lifecycle. Do debits equal credits in all paths?
5. **Test Coverage Audit** — What's tested, what's not. N+1 risks, missing state transitions in tests.
6. **Performance Audit** — N+1 queries, unnecessary preloads, missing indexes, pagination gaps.
7. **Security/Integrity Audit** — Input validation, auth/authz, multi-tenant scoping.

### Fallback to Triage

If the issue turns out to be simple (after fetching context), audit falls back to triage format. No wasted depth on easy problems.

### Surfacing Bugs

In practice, audit on complex issues has surfaced bugs the team hadn't noticed:
- The `Bill.TotalAdjustments` orphaned field (never updated in production code)
- Commission truncation bug (integer math on percentage × invoice total)

## Output: Phased Implementation Plan

Audit ends with an ordered implementation plan where each task is scoped to be its own GitHub issue:

```
## Implementation Plan

### Phase 1: Prep (low risk, can merge independently)
- [ ] Task A: [description] — affects [files]
- [ ] Task B: [description] — affects [files]

### Phase 2: Core Changes (medium risk)
- [ ] Task C: [description] — depends on A, B
- [ ] Task D: [description] — accounting event changes

### Phase 3: Cleanup (after Phase 2 proven in production)
- [ ] Task E: [description]
```

### Splitting Issues

If audit determines the issue is too large, it explicitly says so and describes how to break it into child issues. The team creates those child issues manually, then triages/implements each independently.

## The Conversational Loop

Audit is designed for iteration:
1. Run `@claude /audit`
2. Read the output. Post follow-up questions or decisions in comments.
3. Re-run `@claude /audit` — it reads all prior comments and refines the analysis.
4. Repeat until the implementation plan feels right.

This conversational loop (on issue #136) demonstrated genuine collaborative analysis: the team asked targeted follow-up questions and got refined answers that materially changed the implementation plan.

## Max Turns: 30

Audit runs with `--max-turns 30` (vs triage at 20). This is necessary — a thorough audit of a 25+ file codebase was hitting the 15-turn limit (the original value) 60% through the analysis. The increase to 30 resolved this.
22:T830,# Triage Skill: Readiness Signal Pattern

## The Pattern

Every `/triage` output ends with a readiness signal — a traffic-light assessment of whether the issue is ready to implement:

- **🟢 Green** — Clear scope, low risk, localized changes, existing tests cover the area. Implement immediately.
- **🟡 Yellow** — Moderate uncertainty, touches shared logic, edge cases unclear. Review triage output first, possibly run `/audit`.
- **🔴 Red** — Affects accounting/GL, requires migrations, state machine changes, scope too vague. Run `/audit` first; consider splitting the issue.

## Why It Works

Without a readiness signal, developers tend to start implementing before the scope is clear. The traffic light forces an explicit decision gate: "Are we confident enough to proceed?" This stopped a pattern of implementation work starting on under-scoped or too-large issues.

## Triage Output Structure

```
## Affected Areas
- Models: [list]
- Files: [list]
- State Transitions: [if any]
- Risk Areas: [if any]

## Proposed Approach
1. [Concrete bullet]
2. [Concrete bullet]
...

## Testing Strategy
[What tests to write/update]

## Related Issues
[Links if any]

## Open Questions
[Blockers or important unknowns]

## Readiness: 🟢 Green
[One-line rationale]
```

## Escalation Path

- Green → go straight to `/implement`
- Yellow → discuss in comments, possibly `/audit`, then `/implement`
- Red → always run `/audit`; if audit finds it's too large, break into smaller issues, then triage/implement each

## Max Turns

The triage skill runs with `--max-turns 20`. This is enough for most issues (scoped analysis of 3-10 files). If triage hits the limit, it's a signal the issue is too large and should be escalated to `/audit` (which runs at `--max-turns 30`).

## Contrast with Audit

Triage is **prescriptive** — it identifies the problem and proposes a specific approach. Audit is **exploratory** — it surfaces trade-offs and options without prescribing. Use triage by default; escalate to audit only when triage returns red or the issue is explicitly architectural.

## Permission constants

Permissions are UUID strings organized by resource, defined in `shared-types`:

```typescript
export const perms = {
  invoices: {
    create: "0cde0c13-e5af-40eb-b7df-957620a65043",
    read:   "a33c8e35-2c79-45c8-9f02-b5aace85db22",
    update: "79f4bc16-b8d9-4293-9ffe-9835f9c37860",
    delete: "b1f8c0bb-df00-43db-9e96-0952b9709f63",
    listPageRead: "fc424c5f-60a7-4005-aab7-f95ebd00d17d",
  },
  comments: {
    create: "53fd1293-...",
    // ...
  },
}
```

Using UUIDs (vs. string slugs) means renaming a resource doesn't invalidate existing DB rows.

## Role composition

Base groups bundle related perms, composite groups bundle base groups:

```typescript
const invoicesBaseGroup = {
  perms: [perms.invoices.create, perms.invoices.read, ...],
}

// Composite roles include relevant base groups
const partners = [...invoicesBaseGroup, ...projectsBaseGroup, ...]
const superuser = [...partners, ...adminBaseGroup, ...]
```

## canAccess in resolvers

Every resolver calls `canAccess` as its first line:

```typescript
export const fn = async ({ context, args }) => {
  await canAccess(context, [perms.invoices.update])
  // proceeds only if user has the permission
}
```

`canAccess` accepts an array — passing multiple perms means "any one of these" (OR logic).

## Impersonation support

The system distinguishes `context.user` (authenticated user) from `context.actAsUser` (impersonated user). `canAccess` first checks if the auth user is allowed to impersonate, then checks the act-as user's permissions.

## Benefits

- UUIDs are stable — safe to rename resources without DB migration
- Consistent `listPageRead` permission separates list access from single-record read
- Composite groups make role assignment simple
- Permission cache (10-min TTL) avoids N+1 DB hits per request
27:T6ac,# Inline Auto-Save Pattern for Form Fields

For forms where each field change should persist immediately (no submit button), this pattern fires a mutation directly in the field's `onChange` handler.

## Core implementation

```typescript
const [isSaving, setIsSaving] = useState(false)
const { runMutation: updateMutation } = useUpdateEntity()

const saveField = useCallback(async (params: Partial<EntityFields>) => {
  if (!id || isSaving) return
  setIsSaving(true)
  try {
    const result = await updateMutation({ variables: { id, ...params } })
    if (result?.success) {
      // Invalidate Apollo cache for related queries
      clearRootQueries(client, ["report_EntityListPage"])
      toast.success("Saved")
      refetchData?.()
    }
  } catch (err: any) {
    toast.error(err?.message || "Failed to save")
  } finally {
    setIsSaving(false)
  }
}, [id, isSaving, updateMutation, client, refetchData])
```

## Field usage

```tsx
<InlineDatePicker
  value={data?.someDate}
  onChange={async (date) => {
    await saveField({ someDate: date })
    // Update local form state without marking it dirty
    form.setValue("someDate", date, { shouldDirty: false })
  }}
  disabled={isLocked || isSaving}
/>
```

## Key details

- **`isSaving` guard**: prevents concurrent saves from the same component
- **`shouldDirty: false`**: keeps RHF from thinking the form has unsaved changes after a successful save
- **Disable inputs while saving**: `disabled={isLocked || isSaving}` prevents race conditions
- **Cache invalidation**: `clearRootQueries` refreshes list pages that show the same data
- **`isLocked` pattern**: separate business logic flag (e.g. invoice is finalized) from save-in-progress flag
28:T7c0,# URL-Synced Filter & Pagination State with nuqs

Each list page has a co-located `params.ts` that owns all filter/sort/pagination state in the URL via `nuqs`.

## File structure

```
src/app/(dashboard)/some-list/
├── params.ts       ← owns all URL state
├── filterBar.tsx   ← reads/writes params
└── page.tsx        ← composes FilterBar + table
```

## params.ts pattern

```typescript
"use client"

import { parseAsArrayOf, parseAsInteger, parseAsJson, parseAsString, useQueryStates } from "nuqs"
import { useMemo } from "react"
import { z } from "zod"

const sortSchema = z.array(z.object({ id: z.string(), desc: z.boolean() }))

export const useParams = () => {
  const [params, setParams] = useQueryStates(
    {
      page: parseAsInteger.withDefault(1),
      searchTerm: parseAsString,
      status: parseAsArrayOf(parseAsString).withDefault(["active"]),
      sort: parseAsJson(sortSchema.parse).withDefault([{ id: "createdAt", desc: true }]),
      limit: parseAsInteger.withDefault(50),
    },
    {
      urlKeys: { limit: "perPage" }, // rename URL key without changing JS name
    },
  )

  const updatedParams = useMemo(() => ({
    ...params,
    searchTerm: params.searchTerm ?? undefined, // null → undefined for optional GQL args
    pageIndex: params.page - 1,                 // convert 1-based page to 0-based index
  }), [params])

  return [updatedParams, setParams] as const
}
```

## Key ideas

- **Shareable URLs**: filters survive page refresh and can be copied/bookmarked
- **Type-safe parsers**: `parseAsInteger`, `parseAsString`, `parseAsArrayOf`, `parseAsJson` with Zod validation
- **`.withDefault()`**: guarantees non-null value so downstream code doesn't need null checks
- **`urlKeys`**: rename the URL param without changing the JS variable name
- **null → undefined transform**: nuqs uses `null` for absent values; GraphQL args expect `undefined`
- Returns `[params, setParams]` tuple following React hook convention

```typescript
@ArgsType()
export class QueryArgs {
  @Field(() => String)
  someField: string;
}

export const OutputType = SomeObjectType;

export const fn = async ({
  context,
  args,
}: {
  context: IContext;
  args: QueryArgs;
}): Promise<typeof OutputType> => {
  await canAccess(context, [perms.resource.action]);
  // ...
};
```

Uses `glob` to find all `[mq]_*.ts` files, extracts resolver name from filename (strips `m_`/`q_` prefix), infers GQL type from prefix, then renders a Handlebars template:

```typescript
const getCrudModelResolvers = async (rootDir: string) => {
  const tsFiles = await glob(`${rootDir}/**/[mq]_*.ts`);
  return tsFiles.map((filePath) => {
    const file = path.parse(filePath);
    const gqlType = file.name[0] === 'q' ? 'Query' : 'Mutation';
    return {
      filePath: getFullPathWithoutExtension(filePath),
      resolverName: file.name.slice(2),
      gqlType,
    };
  });
};
```

```handlebars

  constructor(readonly dbService: DbService) {}

{{#each resolvers}}
  @{{gqlType}}(() => {{resolverName}}.OutputType)
  async {{resolverName}}(
    @Args() args: {{resolverName}}.QueryArgs,
    @Context() context: IContext,
  ) {
    return await {{resolverName}}.fn({ context, args });
  }
{{/each}}
}
```

- Adding a new resolver = create one file, run `pnpm generateResolvers`

## Architecture
- Turborepo monorepo with pnpm workspaces
- `packages/db` — shared Drizzle ORM schema, queries, config
- `apps/cli` — Commander.js CLI (`wiki` binary)
- `apps/mcp` — local MCP stdio server (`wiki-mcp` binary)
- `apps/web` — Next.js read-only dashboard

## MCP Server is Just a Local Script
An MCP server does NOT need to be hosted. Claude Desktop and Claude Code spawn it as a local subprocess via stdio. The config just points to a binary:
```json
{ "command": "/absolute/path/to/wiki-mcp" }
```
Always use absolute paths — Claude spawns the process in a clean environment where PATH may not include your shell's bin dirs.

## Registering with Claude Code vs Claude Desktop
- Claude Desktop: `~/Library/Application Support/Claude/claude_desktop_config.json`
- Claude Code: `claude mcp add <name> <absolute-path> --scope user` (writes to `~/.claude.json`)
- These are separate configs — you need to register in both.

## tsup Bundling Gotchas
- All source files get bundled into one `dist/index.js`
- `import.meta.dirname` in the bundle = the `dist/` directory, NOT the original source file's directory
- Path calculations using `import.meta.dirname` must account for this:
  - From `dist/index.js`, `../migrations` → `apps/cli/migrations/` ✓
  - `../../migrations` from `dist/` goes two levels up to `apps/` — wrong
- Mark all runtime deps as `--external` in tsup or they won't resolve at runtime

## pnpm Global Linking
```bash
# Must be run from the package directory
cd apps/cli && pnpm link --global
cd apps/mcp && pnpm link --global
```
- Requires `PNPM_HOME` in PATH (run `pnpm setup` first)
- Creates a shell wrapper script, not a symlink
- `import.meta.dirname` resolves to the actual file path, not the wrapper location

## Workspace Package Resolution in Next.js
- `packages/db` exports TypeScript source directly via `"exports": { ".": "./src/index.ts" }`
- Next.js webpack cannot resolve `.js` imports pointing to `.ts` files
- Fix: use `transpilePackages: ["@ai-wiki/db"]` + webpack `extensionAlias`:
```ts
webpack: (config) => {
  config.resolve.extensionAlias = { '.js': ['.ts', '.tsx', '.js'] };
  return config;
}
```

## drizzle-kit Config Gotcha
- `drizzle.config.ts` is loaded by drizzle-kit's own bundler
- Cannot import from workspace TypeScript packages (no compiled output available)
- Inline any config reading logic directly in `drizzle.config.ts` using Node built-ins only
- `drizzle-kit generate` does NOT need a DB connection — only `migrate` does
- Use a try/catch around `getDatabaseUrl()` so generate works without config

## Programmatic Migrations in a CLI
Don't shell out to `pnpm db:migrate` from a setup command — it breaks when:
1. The working directory is wrong
2. The package is installed globally (no monorepo context)

Instead, use drizzle-orm's migrator directly:
```ts
import { migrate } from "drizzle-orm/postgres-js/migrator";
await migrate(db, { migrationsFolder: join(import.meta.dirname, "../migrations") });
```
Output migration files to `apps/cli/migrations/` so they're co-located with the binary.

## Postgres Connection Pool Hangs Process
The `postgres` driver keeps a connection pool open. CLI commands never exit without:
```ts
program.parseAsync().then(() => process.exit(0));
```
Exception: long-running commands like `serve` should not force-exit.

Store user config at `~/.config/<app>/config.json`, not `.env` in the project dir.
- Works regardless of where the CLI is run from
- Shared between CLI and MCP server automatically
